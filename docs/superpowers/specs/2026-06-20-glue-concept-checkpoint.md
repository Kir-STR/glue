# Glue — концептуальный чекпойнт брейншторма

**Дата:** 2026-06-20
**Статус:** ⛔ ЗАМОРОЖЕН / промотирован. Все косточки (1–13) закрыты; концепция промотирована в **`2026-06-22-glue-concept-design.md`** (источник правды). Этот файл сохраняется как исторический протокол процесса (с адъюдикациями и отвергнутыми вариантами) и **не правится**. При расхождении приоритет за финальным спецом.

## 1. Принятые решения

### Косточка 1 — Хребет модели (Provenance-first)

- Модель **гибридная с привилегированным хребтом**. Цепочка **Решение → Ограничение → Проверка** — главная, но **происхождение ограничения обязательно**.
- **Авторитетный источник** ⊇ { **ADR** (предпочтительный *внутренний*), внешнее требование/политика, унаследованное правило с зафиксированным происхождением }.
- Ограничение **без источника** допустимо **только как временное состояние** с явным статусом и диагностикой (не «orphan» — именно временное состояние).
- **Инвариант:** всякое вето замыкается на авторитетном источнике (не обязательно ADR). Поэтому корректнее называть принцип **Provenance-first**, где ADR — частный (предпочтительный) случай.

### Косточка 2–3 — Онтология артефактов

- **Род: липкий артефакт** (совокупность — «липкие артефакты проекта»; ранее «контекстный артефакт»). «Предписание» — *семантический подтип*: часть артефактов предписывает «как делать», часть только описывает факты (Знания/документация).
- Различение — **фасетами**: ⟨ источник, форма, загрузка, обдумывание, судья, **предмет** ⟩. **Пять** линз-артефактов (Знание / Инструкция / Навык / Решение / Ограничение) = устойчивые комбинации фасетов, а не независимые сущности. **Окружение — не шестая линза, а отдельная ось вердикта** (см. косточка 9). **Предмет** (dev / infra / оба) — ортогональная фасета: линзы предмет-агностичны (см. косточка 9).
- Суть пяти линз (Окружение ниже — как ось, не линза):
  - **Знание** — внешний источник; справочный корпус (llms.txt / документация); *lazy*, в контекст не грузим. Аналогия: книга ПДД.
  - **Правило** (ранее «Инструкция») — внутренний; проза-директива; *eager* (`CLAUDE.md`/`AGENTS.md`/`.claude/rules/*.md`); применение **требует обдумывания**. Аналогия: навыки вождения в голове. Имя «Правило» согласуется с `.claude/rules` invoker; «инструкции» остаётся зонтичным разговорным словом, не линзой.
  - **Навык** — внутренний; **процедура**; *invoked*. Может содержать точки выбора и LLM-рассуждение. Отличительный признак: **можно вызвать и получить ожидаемый результат** (НЕ «делать не думая»).
  - **Решение** — внутренний; выбор + **обоснование**; форма ADR. Сверяет **LLM-судья** на соответствие.
  - **Ограничение** — внутренний; **машиночитаемый предикат**; **кодовый судья**, вето. Становится ограничением **только когда кодовая проверка реализована и подключена** (формализуемость ≠ вето).
  - **Окружение** — **НЕ предписание**; ось «**где**», параметризует вердикт.
- **Что такое Glue:** **дисциплина извлечения и адресуемости** поверх нативных артефактов Claude Code.
  - Берём как есть: Инструкции/Правила (`CLAUDE.md`, rules), Навыки (skills + hooks), Знания (документация — в контекст не пихаем).
  - Извлекаем спрятанное: **Решения → ADR**, **Ограничения → отдельный md + кодовый судья**, **Окружения → спеки инфры + инструкции по работе**.
  - Мотивы извлечения: (1) гигиена контекста; (2) enforceability (нельзя обойти); (3) адресуемость/переиспользование отдельными агентами; (4) трассируемость.
- **Не форсируем** единый тип узла / единое хранение. Разные схемы у ADR/навыка/документа/предиката допустимы (решение уровня реализации).
- **Два мира — модель ИСПОЛНЕНИЯ**, не единственная классификация: интерпретируемый агентом (мягко) ↔ детерминированно контролируемый кодом (жёстко). Сила проверки — **спектр** (интерпретация агентом → LLM-судья → кодовый судья), не бинар. Один артефакт может трогаться обоими режимами (тело навыка читает агент, *запуск* гейтит кодовый судья).
- **Судья — роль в обоих мирах:** LLM-судья (мягко, без вето) ↔ кодовый судья (жёстко, вето). Граница = формализуемость.

### Косточка 4 — Судьи

**Сущности (целевая модель):**

- *Статические (декларативные):*
  - **Ограничение** — машиночитаемый предикат-спека + provenance + приоритет/severity.
  - **Чекер (реализация)** — детерминированный код, вычисляющий предикат (возможно параметризуемый).
  - **Гейт** — точка перехвата (pre-skill-launch / pre-merge / pre-action …).
  - **Назначение** — связь **M:N**: `ограничение × чекер × гейт × env-scope × версия`.
- *Рантайм (события):*
  - **Прогон гейта** — несёт снимок входа.
  - **Результат проверки** — на назначение; состояние `pass | fail | error`.
  - **Итог гейта** — агрегат allow/deny.
- *v1 схлопывает:* 1:1 ограничение↔чекер, один гейт, один результат на прогон. Полный M:N разворачивается при нескольких гейтах/окружениях.

**Выражение предиката — вариант A:**

- Именованный **версионированный чекер** + **строго валидируемый frontmatter** (`checker-id + параметры + env-scope + режим применения`).
- **md** = человеческое описание + provenance (для человека и LLM-судьи).
- Неизвестный чекер / невалидные параметры / несовместимая версия = **ошибка конфигурации**, а не молчаливый пропуск.
- **DSL в v1 не нужен** (carve-out: позже, для простых массовых предикатов; сложное остаётся кодом).
- **Markdown — спецификация только при машиночитаемой структуре.** Свободный текст не исполняется детерминированно.

**Детерминизм — три фазы прогона:**

1. **Сбор доказательств** (грязная фаза, карантин недетерминизма): собрать intent, нормализовать аргументы, снять снимок окружения, прочитать живое состояние **один раз и заморозить** как evidence.
2. **Оценка** (чистая): чекеры над `(версия чекера, версия ограничения, канонический снимок)` → результат.
3. **Агрегация** (чистая): результаты → итог гейта.

- **Вход** = `intent + нормализованные аргументы + снимок окружения + снимок доказательств + версия ограничения + версия чекера`.
- Время/сеть/случайность — **запрещены или передаются версионированным входом**.
- **Строгая формулировка:** одинаковые версии чекера и ограничения + одинаковый канонический снимок входа → одинаковый результат проверки.
- Детерминизм распространяется и на `error`: падение на входе X должно быть воспроизводимым; недетерминированные сбои карантинятся в фазе сбора.

**Полная трасса:** `итог гейта → результаты проверок → прогон → назначение → (чекер + ограничение) → источник`.

**Агрегация — четыре ортогональных понятия (не смешивать):**

| Понятие | Что это | Где живёт |
|---|---|---|
| **severity** | серьёзность нарушения | **двухуровневая:** `baseSeverity` (Ограничение) + `effectiveSeverity` (Назначение); override env-обусловлен, требует обоснования; имена явные, чтобы не скрывать переопределение |
| **enforcement** | `block` / `warn` | на **Назначении**, **env-зависимый**; понижение `block→warn` — явное, с provenance, отдельная валидация (иначе обход защиты) |
| **priority** | порядок выполнения / разрешения конфликтов | **НЕ в v1**; стратегия **deny-overrides**; приоритет **не разрешает нормативные конфликты** |
| **result state** | `pass` / `fail` / `error` | результат прогона (три состояния) |

**Алгоритм прогона гейта:**

1. Отобрать применимые назначения по `гейт × env-scope × intent`.
2. По каждому: снимок → чекер → результат `{pass, fail, error}`.
3. Свёртка `result × enforcement`:
   - `pass` → ок
   - `fail + block` → блокирующее нарушение
   - `fail + warn` → предупреждение
   - `error + block` → **fail-closed** → блокирующее нарушение (причина «checker error»)
   - `error + warn` → предупреждение (ошибку показать) — **fail-open только для warn**
4. **Итог = deny, если есть хоть одно блокирующее нарушение**, иначе allow. К отчёту — все нарушения + предупреждения по severity.

**Провенанс ошибки (`error` не мутный):**

- **config error** (невалидный frontmatter) → ловится при **валидации конфигурации**, набор не грузится.
- **collection failure** (не снят evidence) → рантайм, fail-closed для блокирующих.
- **evaluation error** (чекер падает на валидном снимке) → рантайм, fail-closed для блокирующих + **помечается как дефект чекера**.

**Операционная безопасность загрузки:**

- Отказ загружать невалидный набор — **атомарный**, с сохранением **последней валидной конфигурации** (last-known-good). Невалидный набор не применяется; продолжает действовать предыдущая валидная конфигурация. Иначе одна ошибка frontmatter либо снимает защиту, либо останавливает проект.
- **Нормативные конфликты** выявляются при **валидации / LLM-анализе**, а не разрешаются «выбором победителя» во время опасного действия.

**Связь с существующими правилами проекта:**

- block→warn = «эшелонированная защита» (`operator-gate.md`): обход гейта не каскадируется, требует явной перепроверки.
- версии чекера/ограничения и severity → `versioning.md`.
- петля «LLM-флаг → кандидат → ограничение» → `retro-loop.md`.

### Косточка 5 — Петля промоции

**Один источник, общий с retro-loop.** Сигнал рождается в **разборе только что сделанной работы** («что пошло не так / что сработало»). Плюс отдельный боковой вход — **оператор напрямую** («это в правила» / «это запрещаем»), минующий накопитель и триаж. Различие петель — **не на входе** (адвизор переоценил различие источников), а **на выходе**: какой природы вывод и кто его потребитель.

**Развилка на выходе по типу вывода:**

- вывод «это надо помнить / обдумывать» → **правило** (мягкий мир, читает агент) — короткий хвост;
- вывод «это надо механически не пускать» → **ограничение** (жёсткий мир, исполняет код) — длинный хвост;
- (промежуточная ветка для полноты: «зафиксировать выбор с обоснованием» → ADR/Решение, сверяет LLM-судья — пока не детализируем).

**Два темпа (расцеплены):**

- **Часто (конец сессии):** агент накидывает предложения в накопитель, помечая предполагаемый тип. Дёшево, много, качество разное.
- **Реже (отдельно):** оператор разбирает накопленное. «По итогам работы» описывает только *как предложения попадают в накопитель*; маршрутизация, фильтрация и хвосты — отдельная стадия со своим темпом.

**Накопители раздельные по типу** (правила / ограничения) — обоснование не «разный источник», а **разные мерки на разборе**: для ограничения важна формализуемость и корректность чекера, для правила — нет. (Частично реабилитирует тезис адвизора «два контура», но по другой причине.)

**Трёхуровневая организация накопителя на разборе:**

```
стопка (тип)         ► правила | ограничения
  └ группа ценности  ► важное / в бэклог / отвергнуть   (триаж по тройке review-loop)
      └ сортировка    ► по повторениям (чаще всплывало — выше)
```

- **Повторения (n-count) — сортировка/вес внутри группы ценности, НЕ автозапуск и НЕ привратник.** Подсвечивают важное, ничего не двигают сами. Повторение доказывает только устойчивость проблемы — не правомерность запрета, не формализуемость, не корректность чекера.
- На разборе по каждому предложению — два ортогональных движения: *тип* (стопка; агент предположил, оператор может перекинуть) и *ценность* (триаж).
- Счётчик считает **различимые прецеденты** (разный контекст/вход), не N срабатываний на одном месте.

**Хвост ограничений — два независимых операторских «да»** (по `operator-gate.md`, Gate 1 / Gate 2 split):

- **Gate 1 — принять норму** («это правомерно контролировать»). Нормативное решение; ничего не блокирует. → между гейтами: реализация чекера + перепроверки.
- **Gate 2 — включить вето** («чекер реализован, корректность доказана, подключаем»). Деструктивный/высокорисковый класс — вводится новая блокирующая сила (fail-closed может остановить проект; дыра — пропустить).
- **Gate 2 не наследуется из Gate 1** — на момент G1 чекера ещё нет, его корректность не наблюдаема; предавторизация запрещена. Между гейтами реализация может вскрыть **неформализуемость** → миграция в правило, G2 не наступает.
- Между G1 и G2 — обязательный свежий пакет перепроверок (чекер работает? тесты детерминизма зелёные? **норма не дрейфанула** за время реализации? env-scope?).

**Статусная модель кандидата:** `предложен → принят (G1) → реализован → включён (G2) = Ограничение`; боковые исходы `отклонён` (G1 нет) / `ушёл в правило` (неформализуемо).

**Связь с retro-loop:** общий рисунок эволюции, раздельные контуры; сигнал мигрирует между контурами (retro-наблюдение → кандидат Glue; неформализуемый кандидат Glue → правка правила) **только через оператора, без автоперехода**; обе промоции — через PR.

**Симметрия (зафиксировано, в петлю вверх не входит):** деактивация/откат вето — отдельный деструктивный гейт класса teardown.

### Косточка 6 — Навыки-мутаторы

- **Навык = Claude Code skill** (`SKILL.md`, вызывается по необходимости; гейтинг запуска через хуки — уровень реализации). Механизм хуков не детализируем здесь.
- **Двойственность навыка:** одновременно **артефакт** (тело читает агент) и **действие** (запуск перехватывается гейтом).
- **Зачем Glue держит навыки:** (1) автоматизировать процедуры (вызвал → результат); (2) разгрузить правила — процедура уезжает из «прозы, которую обдумывают» в «вызвал — сработало». Та же граница навык ↔ правило.
- **Owned ↔ observed:**
  - **owned** (свои: `retro`, `save`, …) — Glue поставляет, видит на карте, при необходимости мутирует. Glue сам = «кит из артефактов», навыки — один из видов.
  - **observed** (чужие: superpowers и пр.) — Glue видит как артефакты, **read-only** (не правит), но **их запуск тоже гейтится**.
- **Ключевой синтез: ограничение гейтит ДЕЙСТВИЕ, не навык.** Навык — лишь один из источников действия (то же `push` может прийти из чужого навыка, своего навыка или напрямую от агента). «Гейт накрывает любые навыки» — не отдельное правило, а **следствие**. Согласуется с косточкой 4: ограничение назначается на **гейт (точку перехвата)**, не на навык.
- **Гибрид Glue по отношению к мутациям артефактов:**
  - по умолчанию — **зеркало** (изменение видим постфактум, обновляем карту);
  - для **чувствительных** артефактов — **«дверь»**, т.е. **тот же самый гейт-механизм**. Отдельной конструкции «двери» нет.
  - Два класса действий одного кодового судьи: **«запустить навык»** (гейт на запуске) и **«изменить чувствительный артефакт»** (гейт на мутации, intent вида `mutate constraint`).
  - Формально: нет гейта на действие → зеркало; есть гейт → дверь.
- **Чувствительные артефакты (мутация через гейт):** **ограничения** (ослабить/снять вето — опасно) + **происхождение/provenance и связи источник→ограничение** (подделка подрывает хребет provenance-first).
- **НЕ дублируем гейтом:** **правила** — их изменение идёт через существующий PR + оператор (`retro-loop`). Знания/документация и создание ADR — **зеркало** (но связь «ADR → действующее ограничение» уже чувствительна как provenance).
- **Граница «не агент-фреймворк» держится:** Glue не запускает навыки и не оркестрирует — хранит, показывает, поставляет свои, может **запретить** запуск через кодового судью. Запрет ≠ оркестрация.

### Косточка 7 — Типология рёбер

- **Главная ось — «несущесть»** (участвует ли ребро в вердикте/трассе), не плоский список типов. Три класса:
  - **Несущие (load-bearing)** — участвуют в вердикте и трассе; если врут/пропали — вето нелегитимно или не срабатывает.
  - **Справочные (informational)** — ориентир для человека/агента, на вердикт не влияют.
  - **Структурные (structural)** — физика репозитория.
- **Класс ребра определяет чувствительность** (замыкание с косточкой 6): **несущие — под гейтом** (мутация = чувствительное действие), справочные и структурные — **зеркало** (постфактум).
- **Несущие:**
  - **`derives_from` (источник → ограничение)** — provenance вето, по нему замыкается право блокировать.
  - **`назначение` / `enforces`** — **узел-связка (овеществлённое ребро)**, не простая стрелка: M:N `ограничение × чекер × гейт × env-scope × версия` (косточка 4).
  - **`applies-in-env`** — **НЕ отдельное несущее ребро**; `env-scope` — измерение внутри назначения. Отдельная стрелка только если окружение связывается с чем-то помимо назначения.
- **Справочные:**
  - **`references` / see-also** — упоминание без гарантий.
  - **`promoted_from` (lineage / история эволюции)** — «как артефакт появился» (наблюдение → кандидат → ограничение, косточка 5); аудит прошлого.
  - **`supersedes` (замена)** — «новое ограничение пришло на смену старому» (косточка 13); справочное/lineage, новое→старое. Не путать с `promoted_from` (как появился) и `derives_from` (вето сейчас).
- **Структурные:**
  - **`contains` / part-of** — файл содержит правило.
- **Ключевое различение двух «происхождений» (не сливать в один `derives_from`):**
  - **`derives_from`** = provenance вето, «на каком источнике стоит право блокировать **сейчас**» — **несущее**, под гейтом;
  - **`promoted_from`** = lineage, «как **появился**» — **справочное/аудиторское**, зеркало.
  - Вето легитимно и без знания своей истории; путать «легитимность сейчас» и «как дошли» нельзя.

### Косточка 13 — Петля промоции: жизненный цикл ограничения и cleanup (хвостики косточки 5)

**Хвостик А — статусы (YAGNI на расширение пути кандидата):**

- **Путь кандидата НЕ расширяем:** `предложен → принят → реализован → включён`, + боковые `отклонён` / `ушёл в правило`.
- **Живое ограничение ведёт отдельный жизненный цикл** под опасным (teardown) гейтом, состояния: **работает / приостановлено / снято**.
- **«Заменено» — НЕ состояние, а связь.** Старое уходит в `снято`; новое несёт ребро **`supersedes` → старое**. Иначе не отличить «снято без замены» от «заменено конкретным ограничением».
  - `supersedes` — **справочное / lineage ребро** (косточка 7), сосед `promoted_from` / `references`, **НЕ несущее**. Не путать: `supersedes` (новое→старое, «заменило») ≠ `promoted_from` (артефакт→его кандидат-исток, «как появился») ≠ `derives_from` (источник вето сейчас).

**Хвостик Б — cleanup (убрать из накопителя ≠ уничтожить след):**

- После промоции кандидат **исчезает из рабочего накопителя**, но **не уничтожается бесследно**: остаётся **минимальный неизменяемый след** (идентификатор кандидата, исходное содержание/снимок, источник, итог промоции) — как **разрешаемая цель `promoted_from`**. Иначе `promoted_from` указывал бы на несуществующий объект → оборванное происхождение.
  - След живёт **при новом артефакте либо как отдельная квитанция промоции**, не в рабочем накопителе. Неизменяемый (append-only, по духу audit-trail проекта).
- **Очистка — только после** (1) создания артефакта, (2) записи происхождения, (3) успешной проверки результата. Не раньше — иначе полупровалившаяся промоция теряет кандидата.

### Косточка 12 — Конфликт-детекция

- **Принцип (из косточки 4, добит):** нормативный конфликт **выявляется заранее и показывается оператору**, не разрешается «выбором победителя» на лету.
- **Два слоя детекции по типу конфликта:**
  - **Машинный** — два *ограничения* с чёткими предикатами противоречат в одной ситуации / `env-scope`. Ловится **при валидации/загрузке набора**, детерминированно. Конфликт **блокирует активацию нового набора**; Glue продолжает на **последнем валидном** (last-known-good, косточка 4). Стартовый статус — **подтверждён** (факт по построению).
  - **Смысловой** — две *прозы-правила* противоречат только по смыслу, машинно не видно. Ловит **LLM-анализ**, как **подозрение**, не факт и не вето. Стартовый статус — **подозрение**; оператор подтверждает или отклоняет.
  - **Способ обнаружения задаёт стартовый статус** (машинный → подтверждён; LLM → подозрение).
- **Конфликт — отдельный адресуемый объект** (не просто ребро). Несёт: **область пересечения** (`гейт × env-scope × intent`), **доказательство**, **способ обнаружения**, **статус** (`подозрение → подтверждён → разрешён / отклонён`), **ссылку на решение**. Согласуется с реификацией **назначения** (косточка 4/7): ребро со структурой овеществляем в узел — тот же приём, не ad hoc.
  - На карте показывается компактно как ребро **«спорит с»** между правилами — но это лишь визуальное представление объекта, не сам объект.
- **Явно заданное исключение / приоритет — НЕ конфликт.** Override легитимен, если у него есть **свой артефакт-решение** («A важнее B здесь») с provenance. Конфликт = противоречие **без** объясняющего артефакта. Гасит ложные срабатывания детектора; замыкается на **provenance-first** (норма без источника подозрительна).
- **deny-overrides — аварийное поведение исполнения, не закрытие конфликта.** Подстраховка на лету (побеждает запрет), но конфликт остаётся **открытым** до разрешения.
- **Разрешение — только** изменением артефактов либо явным решением оператора. **Автоматический выбор победителя запрещён.**

### Косточка 11 — Гранулярность правил и структурирующий навык

- **Гранулярность — серединка:** **файл = узел, отдельное правило / раздел = под-узел.** Грубые связи цепляются к файлу, точные (provenance, `derives_from`) — к под-узлу-пункту. Касается в основном **Правил** и **Знаний** (большие тексты с множеством пунктов); у Ограничений / Решений / Навыков одна вещь = один файл, проблемы нет.
- **Адресуемость под-узлов опирается на контролируемую структуру документа**, которую держит **owned структурирующий навык** (умная реорганизация правил/знаний: разложить, сгруппировать, проставить якоря). Серединка *работает* именно потому, что структуру держим сами, а не подстраиваемся под свободный markdown.
- **Три ограждения (приняты):**
  1. **Навык предлагает, не молча переписывает.** Реорганизация правил = мутация правил → через **PR + оператор** (косточка 6 / `retro-loop`), не автоприменение. Glue — помощник-структуризатор, не хозяин текста; «работу не захватываем».
  2. **Структура — разметка ради адресности, не переписывание смысла.** Навык ставит стабильные якоря/ID на пунктах (куда «показать пальцем»), но не приглаживает смысл правила. Нюанс прозы, «которую надо обдумать», сохраняется.
  3. **Опираемся на структуру только при валидации.** Структура поехала (ручная правка мимо навыка) → **мягкая деградация** к файл-гранулярности для этого документа, не падение. Тот же приём, что у ограничений (косточка 4: нет валидной структуры — деградируем мягко).
- **Навык — owned** (как `retro` / `save`, косточка 6): Glue приводит себя в порядок собственным инструментом («кит из артефактов»).
- **Постройка самого навыка — реализация/роадмап.** Здесь фиксируем только **принцип** («структуру держим навыком, на неё опираемся»); время постройки отложено вместе с формой продукта.

### Косточка 10 — Нейминг рода и линз

- **Род — «липкий артефакт»** (ед. ч.); совокупность — **«липкие артефакты проекта»**. Имя завязано на бренд Glue (клей держит «приклеенное»). Отвергнуты: «контекстный артефакт» (ложно намекает на загрузку в контекст — часть артефактов lazy), «сопутствующий», «обвязка».
  - Оттенок «липкий» (в русском бывает «неприятно на ощупь / нашлёпка») **рассмотрен и принят** оператором осознанно — для продукта Glue считывается как фишка, не дефект. Не переоткрывать.
  - «Липкие артефакты» — это и канон, и разговорное; отдельного жаргонного прозвища не вводим.
- **Линза «Инструкция» → «Правило».** Слово «инструкция» было перегружено: и линза (проза-директива), и зонтичное «инструкции агенту» (внешний термин Claude Code). Развязка: переименована **линза**; «инструкции» остаётся разговорным зонтиком. Имя «Правило» согласуется с `.claude/rules` invoker и не конфликтует с «Ограничением» (мягкая проза ↔ жёсткий предикат — различение уже зафиксировано косточками 2–4).
- **Пять линз (канон):** Знание, Правило, Навык, Решение, Ограничение. (Окружение — ось, не линза; косточка 9.)
- **Скоуп нейминга — содержание** (документация, карта, глоссарий). Имена-идентификаторы (slug'и веток, ключи YAML/JSON, импорты) — ASCII, отдельная политика `glossary.md`.

### Косточка 9 — Окружение и инфраструктура (предмет vs ось)

> Мотив оператора: разработка не живёт отдельно от инфраструктуры; инфра-работа сначала незаметна, потом разрастается, и несёт свои знания/решения/правила/навыки/ограничения. Glue склеивает артефакты вокруг **и** разработки, **и** инфры — оставляя сам акт (творчество) свободным.

- **Пять линз-артефактов** (Знание / Правило / Навык / Решение / Ограничение) применяются **равноправно к разработке и инфраструктуре**. Инфра — **первоклассный предмет**, не довесок (свои ADR / правила / ограничения / навыки наравне с дев-артефактами).
- **Окружение — отдельная ось вердикта (`env-scope`), не шестая линза.** Снимает overload слова «Окружение»: ось — это `env-scope`; инфра-артефакты — обычные артефакты с фасетой `предмет=infra`.
- **Предмет (dev / infra / оба) — ортогональная фасета/связь, НЕ дублирование онтологии на два изолированных кластера.** Один артефакт может относиться к обоим: правило миграции связывает код, базу и production. Два изолированных кластера такому артефакту не дают места. Предмет **композируется** с `env-scope` (правило миграции: `предмет ∈ {code, infra}` **и** `env-scope=prod`).
- **Декларация vs живое состояние:** **декларативный профиль** окружения **задаёт идентичность** окружения и якорит `env-scope` (дёшево, детерминированно). **Живое состояние НЕ определяет активный профиль** — оно фиксируется снимком в фазе сбора (косточка 4) и служит **доказательством** для **опциональных** инфра-чекеров «реальность ↔ декларация» (напр. «на prod-профиле реальный branch обязан быть `main`»).
- **Граница «Glue не управляет» — симметрична и усилена:** Glue не провиженит/не деплоит, как не пишет код; он склеивает артефакты *вокруг* обеих работ. Дев и инфра — обе «свободная работа», Glue — соединительная ткань. Усиливает инвариант «не агент-фреймворк / не инфра-тул».
- **Итоговая формула (заменяет «5 предписаний + ось»):** пять линз липких артефактов применяются к разработке и инфраструктуре; `env-scope` — отдельная ось вердикта; декларативный профиль задаёт окружение, снимок живого состояния опционально проверяется.

### Косточка 8 — Карта-UI (представления модели)

- Одна каноническая графовая модель (узлы + рёбра косточек 4–7), **два потребителя, два представления**:
  - **Агент — запросный движок.** НЕ визуализация: программный обход графа (применимые ограничения по `intent × env-scope`, трасса вето, provenance). Ответ — **структурированный** (подграф / трасса / узлы), не пиксели.
  - **Человек — навигационная визуализация.** Аудит, навигация, разбор накопителя, импакт-анализ («что сломается, если убрать ADR» = обход несущих рёбер вверх).
- **Термин «карта-UI» = только человеческое представление.** Агентский интерфейс — **API обхода той же модели**, не «ещё один UI».
- **Совместный разбор полотна** передаётся агенту как **выбранный подграф + фильтры + фокусный узел**, не как изображение. Закрывает остаточный вопрос «нужна ли агенту картинка» — нет, даже при совместном фокусе.
- **Человеческое представление: read-only к модели, интерактивное по навигации.** Ложная бинарность «read-only vs интерактив» снята: `read-only` относится к **мутации модели**, интерактив — к **навигации**. Навигационные примитивы: фильтрация по линзам, раскрытие соседей, выбор трассы, импакт-анализ, переход к источнику. «Read-only» ≠ статичная картинка.
- **Синтез (один обходной слой, два клиента):** навигационные примитивы человека = те же запросы движка (раскрытие соседей, трасса, импакт — это обходы графа). Значит человеческая визуализация — **визуальный клиент того же движка**, не параллельный traversal-слой. Иначе агент и человек видят разные трассы.
- **Мутация вынесена за границу карты (v1).** Редактирование из карты не включаем. Карта *может позже инициировать* отдельную команду изменения, но сама мутация проходит обычный контракт + проверку + **гейт косточки 6** (gated workflow). Перетаскивание ребра не должно становиться обходом переписать несущую связь. Формально: карта — **точка входа** в gated-workflow, не поверхность прямого редактирования.

## 3. Правки к `IDEA.md`

- **ADR-first → Provenance-first.** Вето замыкается на авторитетном источнике, не обязательно ADR. (`IDEA.md` §«ADR-first», обратная цепочка «… → Решение».)
- **Трасса богаче трёхзвенной.** Добавлены звенья **Прогон** и **Назначение**: `итог → результаты → прогон → назначение → (чекер + ограничение) → источник`. (§«ADR-first».)
- **«Опасное действие = ограничение с высоким приоритетом»** — уточняется: опасность выражается через `severity` + `enforcement=block` + fail-closed, **а НЕ через priority** (priority не разрешает конфликты). (§«Кодовые судьи».)
- **Окружение** в `IDEA.md` подаётся и как один из 6 артефактов, и как «ось вердикта». Фиксируем (косточка 9): **пять** линз липких артефактов (Знание / Правило / Навык / Решение / Ограничение) применяются к разработке **и** инфраструктуре; **`env-scope` — отдельная ось вердикта** (не шестая линза); **предмет (dev/infra/оба) — ортогональная фасета**, не два изолированных кластера; декларативный профиль задаёт окружение, снимок живого состояния опционально проверяется. (§«Проектные артефакты» vs §«Кодовые судьи».)
- **«Любое ограничение, выразимое детерминированной проверкой, должно реализовываться кодовым судьёй»** — уточнение: **формализуемость ≠ вето**; правило становится ограничением только после реализации + подключения чекера; невыразимое/нереализованное остаётся правилом/кандидатом. (§«Кодовые судьи», §«LLM-судьи».)
- **Детерминизм «при одинаковых входных данных»** — уточняется: вход = **канонический снимок** (`intent + args + env snapshot + evidence snapshot + версии`); недетерминизм карантинится в фазе сбора. (§«Кодовые судьи».)
- **Markdown-ограничение — спецификация только при машиночитаемой структуре** (frontmatter); свободный текст не исполняется детерминированно. (Новое; уточняет §«Кодовые судьи».)
