# Glue — концепция (финальный спец, v1)

**Дата:** 2026-06-22
**Статус:** финальный концепт-спец. Источник правды по концепции Glue.
**Происхождение:** промотирован из `2026-06-20-glue-concept-checkpoint.md` (замороженный рабочий протокол брейншторма «по косточкам»). Чекпойнт сохраняется как исторический артефакт процесса; при расхождении приоритет за этим документом.
**Скоуп:** концепция (что за продукт, его модель и инварианты). **Форма продукта** (CLI / web-граф / слой поверх invoker / протокол) — **вне этого документа**, ведётся отдельным раундом дизайна. Implementation plan сюда не примешивается.

---

## 1. Что такое Glue

**Glue — дисциплина извлечения и адресуемости** поверх нативных артефактов Claude Code. Он не управляет кодом, не заменяет инструменты разработки / планирования / таск-трекинга и не оркеструет агентов. Glue хранит, связывает и показывает **артефакты вокруг работы** — оставляя саму работу (творчество — и разработку, и инфраструктуру) свободной.

Цель — сделать структуру проекта и его ограничения **наблюдаемыми, управляемыми и трассируемыми**, с приоритетом происхождения (provenance-first).

**Границы (что Glue НЕ делает):**

- не пишет код и не провиженит/деплоит инфраструктуру;
- не запускает навыки и не оркеструет процессы;
- может **запретить** действие через кодового судью — но запрет ≠ оркестрация.

**Что Glue берёт как есть vs извлекает:**

- **Берёт как есть:** Правила (`CLAUDE.md`, `.claude/rules`), Навыки (skills + hooks), Знания (документация — в контекст не пихаем).
- **Извлекает спрятанное:** Решения → ADR; Ограничения → отдельный md + кодовый судья; инфра-знание → спеки + правила работы.
- **Мотивы извлечения:** (1) гигиена контекста; (2) enforceability (нельзя обойти); (3) адресуемость / переиспользование отдельными агентами; (4) трассируемость.

---

## 2. Хребет модели — Provenance-first

Модель **гибридная с привилегированным хребтом**. Главная цепочка — **Решение → Ограничение → Проверка**, но **происхождение ограничения обязательно**.

- **Авторитетный источник** ⊇ { **ADR** (предпочтительный *внутренний* случай), внешнее требование / политика, унаследованное правило с зафиксированным происхождением }.
- **Инвариант:** всякое вето замыкается на авторитетном источнике (не обязательно ADR). Поэтому принцип — **provenance-first**, где ADR — частный (предпочтительный) случай.
- Ограничение **без источника** допустимо **только как временное состояние** с явным статусом и диагностикой (не «orphan» — именно временное состояние).

**Полная трасса вето:** `итог гейта → результаты проверок → прогон → назначение → (чекер + ограничение) → источник`.

## 3. Онтология: липкие артефакты

**Род — «липкий артефакт»** (совокупность — «липкие артефакты проекта»). Имя завязано на бренд Glue: клей держит «приклеенное» рядом с работой и связывает между собой.

«Предписание» — *семантический подтип*: часть артефактов предписывает «как делать», часть только описывает факты (Знания).

**Различение — фасетами:** ⟨ источник, форма, загрузка, обдумывание, судья, предмет ⟩. Линзы — устойчивые комбинации фасетов, а не независимые сущности.

### Пять линз (канон)

| Линза | Источник | Форма | Загрузка | Судья / контроль |
|---|---|---|---|---|
| **Знание** | внешний | справочный корпус (llms.txt / документация) | *lazy* (в контекст не грузим) | — |
| **Правило** | внутренний | проза-директива (`CLAUDE.md` / `.claude/rules`) | *eager* | требует обдумывания агентом |
| **Навык** | внутренний | процедура (skill) | *invoked* | можно вызвать и получить результат |
| **Решение** | внутренний | выбор + обоснование (ADR) | — | LLM-судья (соответствие) |
| **Ограничение** | внутренний | машиночитаемый предикат | — | кодовый судья, **вето** |

- **Правило** (ранее «Инструкция»): имя согласовано с `.claude/rules`. «Инструкции» остаётся зонтичным разговорным словом, не линзой. Не конфликтует с «Ограничением»: мягкая проза ↔ жёсткий предикат.
- **Ограничение** становится ограничением **только когда кодовая проверка реализована и подключена** (формализуемость ≠ вето).
- **Не форсируем** единый тип узла / единое хранение: у ADR / навыка / документа / предиката могут быть разные схемы (решение уровня реализации).

### Окружение — ось, не линза

**Окружение — отдельная ось вердикта (`env-scope`)**, а не шестая линза. Ось «где» параметризует вердикт (одно ограничение блокирует на prod, предупреждает на local).

### Предмет — ортогональная фасета (разработка / инфраструктура)

Пять линз применяются **равноправно к разработке и инфраструктуре**. Инфра — **первоклассный предмет**, не довесок: у неё свои ADR / правила / ограничения / навыки наравне с дев-артефактами.

- **Предмет (dev / infra / оба) — ортогональная фасета**, НЕ дублирование онтологии на два изолированных кластера. Один артефакт может относиться к обоим: правило миграции связывает код, базу и production.
- Предмет **композируется** с `env-scope`: правило миграции — `предмет ∈ {dev, infra}` **и** `env-scope=prod`.
- **`env-scope` якорится декларативным профилем** окружения (дёшево, детерминированно). **Живое состояние НЕ определяет активный профиль** — оно снимается снимком (см. §5) и служит доказательством для **опциональных** инфра-чекеров «реальность ↔ декларация».

### Скоуп нейминга

Имена линз/рода — это нейминг **содержания** (документация, карта, глоссарий). Имена-идентификаторы (slug'и веток, ключи YAML/JSON, импорты) — ASCII, отдельная политика (`glossary.md`).

## 4. Два мира исполнения — спектр судей

**Два мира — модель ИСПОЛНЕНИЯ**, не единственная классификация:

- **интерпретируемый агентом** (мягко) ↔ **детерминированно контролируемый кодом** (жёстко).
- Сила проверки — **спектр**: интерпретация агентом → LLM-судья → кодовый судья. Не бинар.
- Один артефакт может трогаться обоими режимами: тело навыка читает агент, *запуск* гейтит кодовый судья.

**Судья — роль в обоих мирах:** LLM-судья (мягко, рекомендательно, без вето) ↔ кодовый судья (жёстко, вето). Граница = **формализуемость**. Повторяющийся LLM-флаг → кандидат в ограничение (см. §9).

## 5. Судьи и проверки

### Сущности (целевая модель)

*Статические (декларативные):*

- **Ограничение** — машиночитаемый предикат-спека + provenance + severity (`baseSeverity`). (priority — **не в v1**, см. таблицу агрегации.)
- **Чекер (реализация)** — детерминированный код, вычисляющий предикат (возможно параметризуемый).
- **Гейт** — точка перехвата (pre-skill-launch / pre-merge / pre-action …).
- **Назначение** — связь **M:N**: `ограничение × чекер × гейт × env-scope × версия`.

*Рантайм (события):*

- **Прогон гейта** — несёт снимок входа.
- **Результат проверки** — на назначение; состояние `pass | fail | error`.
- **Итог гейта** — агрегат allow/deny.

*v1 схлопывает:* 1:1 ограничение↔чекер, один гейт, один результат на прогон. Полный M:N разворачивается при нескольких гейтах/окружениях.

### Выражение предиката

- Именованный **версионированный чекер** + **строго валидируемый frontmatter** (`checker-id + параметры + env-scope + режим применения`).
- **md** = человеческое описание + provenance (для человека и LLM-судьи).
- Неизвестный чекер / невалидные параметры / несовместимая версия = **ошибка конфигурации**, а не молчаливый пропуск.
- **Markdown — спецификация только при машиночитаемой структуре.** Свободный текст не исполняется детерминированно.
- **DSL в v1 не нужен** (carve-out: позже, для простых массовых предикатов; сложное остаётся кодом).

### Детерминизм — три фазы прогона

1. **Сбор доказательств** (грязная фаза, карантин недетерминизма): собрать intent, нормализовать аргументы, снять снимок окружения, прочитать живое состояние **один раз и заморозить** как evidence.
2. **Оценка** (чистая): чекеры над `(версия чекера, версия ограничения, канонический снимок)` → результат.
3. **Агрегация** (чистая): результаты → итог гейта.

- **Вход** = `intent + нормализованные аргументы + снимок окружения + снимок доказательств + версия ограничения + версия чекера`.
- Время / сеть / случайность — **запрещены или передаются версионированным входом**.
- **Строгая формулировка:** одинаковые версии чекера и ограничения + одинаковый канонический снимок → одинаковый результат. Детерминизм распространяется и на `error`.

### Агрегация — четыре ортогональных понятия

| Понятие | Что это | Где живёт |
|---|---|---|
| **severity** | серьёзность нарушения | двухуровневая: `baseSeverity` (Ограничение) + `effectiveSeverity` (Назначение); override env-обусловлен, требует обоснования, имена явные |
| **enforcement** | `block` / `warn` | на Назначении, env-зависимый; понижение `block→warn` — явное, с provenance, отдельная валидация |
| **priority** | порядок выполнения / разрешения конфликтов | **НЕ в v1**; стратегия **deny-overrides**; приоритет **не разрешает нормативные конфликты** |
| **result state** | `pass` / `fail` / `error` | результат прогона |

### Алгоритм прогона гейта

1. Отобрать применимые назначения по `гейт × env-scope × intent`.
2. По каждому: снимок → чекер → результат `{pass, fail, error}`.
3. Свёртка `result × enforcement`:
   - `pass` → ок
   - `fail + block` → блокирующее нарушение
   - `fail + warn` → предупреждение
   - `error + block` → **fail-closed** → блокирующее нарушение (причина «checker error»)
   - `error + warn` → предупреждение (fail-open только для warn)
4. **Итог = deny, если есть хоть одно блокирующее нарушение**, иначе allow. К отчёту — все нарушения + предупреждения по severity.

### Провенанс ошибки

- **config error** (невалидный frontmatter) → ловится при валидации конфигурации, набор не грузится.
- **collection failure** (не снят evidence) → рантайм, fail-closed для блокирующих.
- **evaluation error** (чекер падает на валидном снимке) → рантайм, fail-closed для блокирующих + помечается как дефект чекера.

### Операционная безопасность загрузки

- Отказ загружать невалидный набор — **атомарный**, с сохранением **последней валидной конфигурации** (last-known-good). Невалидный набор не применяется; продолжает действовать предыдущий валидный.
- **Нормативные конфликты** выявляются при валидации / LLM-анализе (см. §10), а не разрешаются «выбором победителя» во время опасного действия.

## 6. Карта: одна модель, два представления

Одна каноническая графовая модель (узлы + рёбра, §3/§5/§7) — **два потребителя, два представления:**

- **Агент — запросный движок.** НЕ визуализация: программный обход графа (применимые ограничения по `intent × env-scope`, трасса вето, provenance). Ответ — **структурированный** (подграф / трасса / узлы), не пиксели.
- **Человек — навигационная визуализация.** Аудит, навигация, разбор накопителя, импакт-анализ («что сломается, если убрать ADR» = обход несущих рёбер вверх).

**Решения:**

- Термин «карта-UI» = **только человеческое представление**. Агентский интерфейс — API обхода той же модели, не «ещё один UI».
- Совместный разбор передаётся агенту как **выбранный подграф + фильтры + фокусный узел**, не как изображение.
- Человеческое представление: **read-only к модели, интерактивное по навигации** (фильтрация по линзам, раскрытие соседей, выбор трассы, импакт-анализ, переход к источнику). «Read-only» ≠ статичная картинка.
- **Один обходной слой, два клиента:** навигационные примитивы человека = те же запросы движка. Человеческая визуализация — визуальный клиент того же движка, не параллельный traversal-слой.
- **Мутация вынесена за границу карты (v1):** редактирование из карты не включаем. Карта может позже *инициировать* отдельную команду изменения, но сама мутация проходит обычный контракт + проверку + гейт (§8) — gated workflow, не прямое редактирование полотна.

## 7. Типология рёбер — по несущести

**Главная ось — «несущесть»** (участвует ли ребро в вердикте/трассе), не плоский список типов. **Класс ребра определяет чувствительность** (см. §8): несущие — под гейтом, прочие — зеркало.

**Несущие (load-bearing)** — участвуют в вердикте и трассе; если врут/пропали — вето нелегитимно или не срабатывает.

- **`derives_from` (источник → ограничение)** — provenance вето, по нему замыкается право блокировать **сейчас**.
- **`назначение` / `enforces`** — **узел-связка (овеществлённое ребро)**: M:N `ограничение × чекер × гейт × env-scope × версия` (§5).
- `applies-in-env` — **НЕ отдельное несущее ребро**; `env-scope` — измерение внутри назначения.

**Справочные (informational)** — ориентир для человека/агента, на вердикт не влияют.

- **`references` / see-also** — упоминание без гарантий.
- **`promoted_from` (lineage)** — «как артефакт появился» (наблюдение → кандидат → ограничение, §9); аудит прошлого.
- **`supersedes` (замена)** — «новое ограничение пришло на смену старому» (§9), новое→старое.

**Структурные (structural)** — физика репозитория.

- **`contains` / part-of** — файл содержит правило.

**Не сливать три «происхождения»:** `derives_from` (вето сейчас, несущее) ≠ `promoted_from` (как появился, справочное) ≠ `supersedes` (что заменило, справочное). Вето легитимно и без знания своей истории.

## 8. Навыки-мутаторы и чувствительные действия

- **Навык = Claude Code skill** (`SKILL.md`, вызывается по необходимости; гейтинг запуска через хуки — уровень реализации).
- **Двойственность навыка:** одновременно **артефакт** (тело читает агент) и **действие** (запуск перехватывается гейтом).
- **Owned ↔ observed:**
  - **owned** (свои: `retro`, `save`, структурирующий навык §9, …) — Glue поставляет, видит на карте, при необходимости мутирует. Glue сам = «кит из артефактов».
  - **observed** (чужие: superpowers и пр.) — Glue видит как артефакты, **read-only**, но **их запуск тоже гейтится**.
- **Ключевой синтез: ограничение гейтит ДЕЙСТВИЕ, не навык.** Навык — лишь один из источников действия (то же `push` может прийти из чужого навыка, своего навыка или напрямую от агента). Ограничение назначается на **гейт (точку перехвата)**, не на навык.
- **Гибрид «зеркало vs дверь»:**
  - по умолчанию — **зеркало** (изменение видим постфактум, обновляем карту);
  - для **чувствительных** артефактов — **«дверь»** = тот же гейт-механизм (отдельной конструкции «двери» нет). Нет гейта на действие → зеркало; есть гейт → дверь.
  - Два класса действий одного кодового судьи: «запустить навык» (гейт на запуске) и «изменить чувствительный артефакт» (гейт на мутации, intent вида `mutate constraint`).
- **Чувствительные артефакты:** **ограничения** (ослабить/снять вето — опасно) + **provenance и связи источник→ограничение** (подделка подрывает хребет).
- **НЕ дублируем гейтом:** **правила** — их изменение идёт через существующий PR + оператор (`retro-loop`). Знания/документация и создание ADR — зеркало (но связь «ADR → действующее ограничение» уже чувствительна как provenance).
- **Класс ребра → чувствительность:** несущие рёбра (§7) — под гейтом; справочные и структурные — зеркало.

## 9. Структура документов и петля промоции

### Гранулярность правил

- **Серединка:** **файл = узел; под-узел материализуется лениво** — когда отдельное смысловое правило нуждается в собственной связи, области применения, происхождении или проверке. Раздел Markdown **сам по себе не обязан** быть узлом. Грубые связи — к файлу, точные (provenance, `derives_from`) — к под-узлу. Касается в основном **Правил** и **Знаний**; у Ограничений / Решений / Навыков одна вещь = один файл.
- **Адресуемость под-узлов опирается на контролируемую структуру**, которую держит **owned структурирующий навык** (умная реорганизация правил/знаний: разложить, сгруппировать, проставить якоря).
- **Три ограждения:**
  1. **Навык предлагает, не молча переписывает** — мутация правил через PR + оператор; Glue не хозяин текста.
  2. **Структура — разметка ради адресности, не переписывание смысла** (стабильные якоря/ID; нюанс прозы сохраняется).
  3. **Опираемся только при валидации** — структура поехала → мягкая деградация к файл-гранулярности **для навигации** (справочные/структурные ссылки), не падение. Но **несущая ссылка** (`derives_from` на под-узел, §7) к неразрешимому точному источнику — это **ошибка конфигурации + last-known-good** (§5), а не молчаливое расширение до файла: иначе провенанс вето тихо огрубляется и вето теряет легитимность.
- Постройка самого навыка — реализация/роадмап; здесь фиксируем только принцип.

### Петля промоции

**Один источник, общий с retro-loop:** сигнал рождается в **разборе только что сделанной работы** + боковой вход «оператор напрямую». Различие петель — не на входе, а **на выходе** (природа вывода и его потребитель).

**Развилка на выходе:**

- «надо помнить / обдумывать» → **правило** (мягкий мир) — короткий хвост;
- «надо механически не пускать» → **ограничение** (жёсткий мир) — длинный хвост;
- «зафиксировать выбор с обоснованием» → ADR/Решение (LLM-судья).

**Два темпа (расцеплены):** часто (конец сессии) агент накидывает предложения в накопитель с предполагаемым типом; реже (отдельно) оператор разбирает.

**Накопители раздельные по типу** (правила / ограничения) — разные мерки на разборе (для ограничения важна формализуемость и корректность чекера).

**Трёхуровневая организация на разборе:** стопка (тип) → группа ценности (важное / в бэклог / отвергнуть) → сортировка по повторениям. **Повторения (n-count) — вес внутри группы, НЕ автозапуск и НЕ привратник.**

**Хвост ограничений — два независимых операторских «да»** (Gate 1 / Gate 2 split, `operator-gate`):

- **Gate 1 — принять норму** («это правомерно контролировать»). Ничего не блокирует. → между гейтами: реализация чекера + перепроверки.
- **Gate 2 — включить вето** («чекер реализован, корректность доказана, подключаем»). Деструктивный/высокорисковый класс.
- **Gate 2 не наследуется из Gate 1.** Между гейтами реализация может вскрыть неформализуемость → миграция в правило, G2 не наступает.

**Статусная модель кандидата:** `предложен → принят (G1) → реализован → включён (G2) = Ограничение`; боковые исходы `отклонён` / `ушёл в правило`. **Путь кандидата не расширяем** (YAGNI).

**Жизненный цикл живого ограничения** (отдельно, под teardown-гейтом): **работает / приостановлено / снято**. «Заменено» — **не состояние, а ребро `supersedes`** (§7).

**Cleanup после промоции:** кандидат **исчезает из рабочего накопителя**, но остаётся **минимальный неизменяемый след** (id, снимок содержания, источник, итог) как разрешаемая цель `promoted_from` — иначе оборванное происхождение. След живёт при новом артефакте или как квитанция промоции (append-only). **Очистка — только после** создания артефакта + записи происхождения + проверки результата.

**Связь с retro-loop:** общий рисунок, раздельные контуры; миграция сигнала между контурами **только через оператора**; обе промоции — через PR.

## 10. Конфликт-детекция

**Принцип:** нормативный конфликт **выявляется заранее и показывается оператору**, не разрешается «выбором победителя» на лету.

**Два слоя по типу конфликта:**

- **Машинный** — два ограничения с чёткими предикатами противоречат в одной ситуации / `env-scope`. Ловится при валидации/загрузке набора, детерминированно. **Блокирует активацию нового набора**; Glue продолжает на last-known-good. Стартовый статус — **подтверждён**.
- **Смысловой** — две прозы-правила противоречат только по смыслу. Ловит LLM-анализ как **подозрение**, не факт и не вето. Стартовый статус — **подозрение**; оператор подтверждает/отклоняет.
- Способ обнаружения задаёт стартовый статус.

**Конфликт — отдельный адресуемый объект** (не просто ребро): несёт область пересечения (`гейт × env-scope × intent`), доказательство, способ обнаружения, статус (`подозрение → подтверждён → разрешён / отклонён`), ссылку на решение. На карте показывается компактно как ребро **«спорит с»** — но это лишь визуальное представление объекта.

**Явно заданное исключение / явное решение о старшинстве — НЕ конфликт.** Override легитимен, если у него есть свой артефакт-решение с provenance. Конфликт = противоречие **без** объясняющего артефакта (замыкается на provenance-first).

**deny-overrides — аварийное поведение исполнения, не закрытие конфликта.** Конфликт остаётся открытым до разрешения. **Разрешение — только** изменением артефактов либо явным решением оператора; авто-победитель запрещён.

## 11. Связь с правилами проекта

- **`operator-gate`:** Gate 1 / Gate 2 split (петля промоции, мутация чувствительного); block→warn = эшелонированная защита (обход не каскадируется).
- **`versioning`:** версии чекера/ограничения; **изменение `severity` входит в версию ограничения** (severity — часть спеки ограничения).
- **`retro-loop`:** петля «LLM-флаг → кандидат → ограничение»; общий рисунок эволюции, раздельные контуры.
