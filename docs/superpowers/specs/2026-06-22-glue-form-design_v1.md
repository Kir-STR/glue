# Glue — форма продукта (design, v1)

**Дата:** 2026-06-22
**Статус:** ⚠️ **ЗАМЕНЁН** на `2026-06-22-glue-form-design_v2.md`. Исторический артефакт. Исходил из модели «защита от обхода» (язык TCB / актор-угроза / четырёхвектор / break-glass); оператор сузил модель до «помогаем избегать честных ошибок, не защищаем от намеренного обхода» → `v2`. Не использовать как источник правды.
**Статус (исходный):** рабочий design формы. Ведётся отдельным раундом поверх застывшей концепции (`2026-06-22-glue-concept-design.md`). Концепцию не переписывает; при расхождении по **концепции** приоритет за концепт-спецом, по **форме** — за этим документом.
**Скоуп:** форма продукта Glue — чем Glue является в момент исполнения, граница доверия, упаковка ядра, клиенты. Концептуальная модель (линзы, рёбра, судьи, петля промоции) — в концепт-спеце, здесь не дублируется.
**Происхождение:** раунд брейншторма «форма продукта», под-косточка 1 (корневая формула + логическая карта доверия). Метод — проза, по одному решению, взаимная адъюдикация.

## 1. Корневая формула — дисциплина с минимальным ядром принуждения

Glue — **дисциплина с минимальным машинным ядром принуждения**. Гибрид «дисциплина + ядро» **выводится** из инварианта «два мира» (концепт-спец §4), а не выбирается из меню «движок / дисциплина / гибрид»: машинный мир требует кода ровно там, где концепция объявила вердикт детерминированным; всё остальное (линзы, навигация, проза-правила, петля промоции) остаётся нативными артефактами Claude Code и соглашениями. В момент жёсткой проверки ядро **является** движком; вне его Glue — конвенция поверх нативного.

### 1.1. Источник правды — файлы; состояние ядра производно

Канон — **файлы проекта**. Ядро **не заводит вторую авторитетную БД**: оно компилирует файлы в валидированный снимок и хранит только **производный** индекс и last-known-good. Состояние ядра целиком пересобираемо из файлов. (Согласовано с концепт-спец §6 «одна каноническая модель» и §5 «last-known-good».)

### 1.2. Семь обещаний минимального ядра

В доверенное ядро входят только машинные обещания концепции:

1. валидация несущей модели и атомарная активация набора;
2. разрешение назначений (`ограничение × чекер × гейт × env-scope × версия`);
3. фиксация снимка доказательств (evidence);
4. запуск кодовых чекеров;
5. агрегация вердиктов и fail-closed;
6. машинная конфликт-детекция (концепт-спец §10, машинный слой);
7. трасса вердикта и last-known-good.

Карта, LLM-анализ, промоция и работа с прозой в ядро **не** входят.

## 2. Граница доверенного контура (TCB)

### 2.1. Критерий членства

TCB — **всё, чья корректность или целостность принимается как доверенная предпосылка гарантии**. Операциональный тест: если тихая компрометация компонента даёт несанкционированный allow **без обнаружения до эффекта** — компонент в TCB. Граница всегда относится к **конкретной гарантии и её модели угроз**, не абсолютна.

- **Чувствительность артефакта** = статическая проекция TCB на **управляемые** артефакты: те, чьё несанкционированное **изменение** меняет смысл / вычисление / доставку / применение вето. Чувствительность наследуется новым звеном enforcement-пути автоматически (критерий, не перечень). Текущая инстанциация enforcement-пути: `перехват → назначение → чекер → ограничение → источник` (+ конфигурация гейта, enforcement, адаптер).
- **Сырой вход недоверен**, даже причинно влияя на вердикт. Гарантия предполагает истинным не реальность, а **снимок того, что увидел доверенный сборщик**.

### 2.2. Enforcement-путь ↔ evidence-путь

На вердикт влияют два пути с разным статусом доверия:

- **enforcement-путь** (управляемая модель) — чувствителен; мутация гейтится.
- **evidence-путь** (наблюдаемая реальность) — не чувствителен как артефакт; не управляется Glue.
  - Снимок обеспечивает **воспроизводимость, не достоверность**: он неизменяем и атрибутирован, гарантирует «честно показываем, что увидел сборщик».
  - **Достоверность** несёт чувствительность **аппарата наблюдения**: сборщик evidence, его конфигурация, канал, доверенные идентификаторы — в TCB. Защищаются двумя механизмами: **гейт мутации** (статика — конфиг/идентификаторы) **и** проверка **происхождения + целостности исполнения + атрибуции результата** (рантайм — provenance-first хребет). Скомпрометированный сборщик может оставить код нетронутым (подмена процесса/зависимости/идентичности/канала) — гейт мутации этого не ловит, поэтому рантайм-проверка обязательна и не подменяется им.

### 2.3. Тупой адаптер

Адаптеры точек перехвата (hooks / CI / git-hooks) — в TCB **по ответственности** (корректно передать событие, проверить получение вердикта, применить без инверсии/подмены), но **без нормативной логики**: всё решение — в ядре. Так доверенная база остаётся минимальной при множестве перехватчиков: размножается перехват, не политика.

### 2.4. Обнаружимость уменьшает TCB — только fail-safe detection

Компонент **выходит** из TCB, лишь если нарушение обнаруживается **до allow/эффекта** и переводит систему в deny. Постфактумный шумный аудит TCB не сокращает (allow уже случился). Предотвращение, не форензика.

## 3. Логическая карта доверия (требования, без реализации)

Карта задаёт **требования** принуждения. Состояния охвата на логическом этапе — строго «**принуждение требуется / не требуется**»; «защищено / незащищено» появляются **только** после наложения фактического покрытия (§5).

### 3.1. Система признаков → конъюнкция обязательств

Действие набирает обязательства от своих признаков; обязательства композируются в **граф с частичным порядком**. Семантика удовлетворённости — **конъюнкция (⋀) всех применимых именованных узлов**; рёбра задают **только порядок**, не импликацию; отсутствие ребра = независимость, не необязательность. Непоглощение (эшелонированная защита) — **следствие конъюнктивной семантики**: ни один гейт не зачитывается за другой (G1 ≠ G2, admin-override одного инварианта не каскадирует, teardown не покрывается outward-гейтом).

| Признак действия | Обязательство |
|---|---|
| ø (default — нечувствительная мутация) | свободная зона: Glue не добавляет своего принуждения; внешние контуры (operator-gate, PR-policy) действуют |
| чтение / навигация / карта / анализ | ø (by design) |
| под ограничением | +enforcing-M (рантайм-судья) |
| outward-facing | +H (operator-gate, буквальное «да») |
| destructive | +H (report+STOP, не UX-модаль) |
| затрагивает enforcement-путь | +enforcing-M (перехват + проверка авторизации) +H (по типу перехода, §3.3); активация набора → +enforcing-M (валидация + атомарная активация) |
| промоция → Правило | +H (норма); advisory-M (структура) — диагностическая аннотация, → enforcing-M только при причинном влиянии на блокировку |
| промоция → Ограничение | разворачивается в «затрагивает enforcement-путь» |

### 3.2. enforcing-M ↔ advisory-M

- **enforcing-M** — fail-closed, блокирует, **в TCB** (семь обещаний ядра); узел графа обязательств.
- **advisory-M** — fail-soft, деградирует, сигналит, **вне TCB** (зона карты/навигации); **отдельная аннотация диагностики, не участвует в расчёте «защищено»**.
- Переход advisory → enforcing управляется единым причинным критерием (§2.1): как только результат начинает использоваться для блокировки — компонент в TCB.
- **H-узел никогда не одинок:** подделываемость авторизации (vector подлинности, §4) делает обязательным **машинного/платформенного держателя подлинности** при каждом H. H без такого держателя — недоказуемое обязательство. Это **целевой инвариант — требование к упаковке** (§6); разговорный operator-gate сам по себе его пока не доказывает.

### 3.3. Фазовый скелет несущей модели

Прямая правка несущего артефакта и промоция → Ограничение разворачиваются в **один** скелет (M инвариантен по типу перехода, человеческие гейты параметризуются):

```
авторизация перехода → перехваченная мутация → M-валидация
  → необходимые предактивационные гейты → атомарная M-активация
```

Конкретные H-гейты и их положение задаёт **тип перехода**:

| Тип перехода | Человеческие гейты |
|---|---|
| создание нового вето | G1 (норма) → G2 (включение) |
| ослабление / приостановка / снятие вето | teardown-гейт |
| техническая замена без смены смысла | authorize-mutation (не G1) |

G2 — **не** обязательная фаза всякого перехода; это частный предактивационный гейт создания нового вето.

## 4. Контроль точки перехвата

Сила гарантии узла определяется **не расположением механизма, а отношением `актор → возможности → держатель`**. Для каждого enforcing-узла — проверяемый **четырёхвектор**: актор заявленной модели угроз не должен мочь:

1. **независимость** — удалить или обойти перехват;
2. **полнота покрытия** — направить действие по эквивалентному неперехваченному пути;
3. **подлинность** — подделать либо повторно использовать вердикт/авторизацию;
4. **поведение при отказе** — превратить отказ канала в разрешение (канал fail-closed).

Классы держателей:

- **локальный класс** (agent-side hook + repo-side git-hook — один класс; путь в репо сам по себе не граница доверия) — **дефолт при отсутствии доказанной внешней границы**; становится сильнее только при отдельном контроле ОС/платформы;
- **платформенный** (CI / branch protection / server-side) — силён **только против актора без административного контроля платформы**;
- **корень доверия** — конечный администратор платформы обычно способен изменить сам механизм; он либо вне модели угроз как корень доверия, либо гарантия против него **не заявляется**.

**Оператор — источник авторизации, не держатель перехвата.** Держатель — машина/платформа, проверяющая **неподделываемое** подтверждение оператора.

---

## 5. Политика локального обхода

Отдельная **явная** политика поверх четырёхвектора, определяется для **пары (enforcing-узел × актор)**, не для узла вообще. Три состояния:

- **запрещён** — для акторов заявленной модели угроз;
- **только break-glass** — санкционированный обход;
- **гарантия не заявляется** — против данного актора.

**Break-glass — отдельный контролируемый путь исполнения, не отключение основного гейта.** Санкционирован, только если независимый держатель обеспечивает авторизацию + свежие проверки + **неподделываемый** audit-trail (совпадает с `operator-gate` admin override + `review-loop` Gate split: свежий гейт каждый раз, деструктивный класс = report+STOP, эшелонированно, обязательный след). Если актор способен обойти и журнал — это **тихий обход**, не escape-hatch.

**Инвариант «тихого обхода нет»:** обход либо запрещён механизмом, либо санкционирован со следом. Неизвестный / непроверенный обход — диагностированное «незащищено» (выход проверки покрытия), **не** четвёртое состояние политики.

---

## 6. Открытые требования к упаковке

Выводимы из карты доверия как требования; удовлетворяются на следующем шаге (выбор упаковки ядра: библиотека / CLI / локальный сервис / протокол + web-визуализация):

- **машинный/платформенный держатель подлинности H** — неподделываемая проверка операторской авторизации (§3.2);
- **fail-safe detection до эффекта** для компонентов, которые мы хотим вывести из TCB (§2.4);
- удовлетворение **четырёхвектора** (§4) и **политики обхода** (§5) выбранным механизмом перехвата;
- сохранение инварианта «файлы — источник правды, состояние ядра производно» (§1.1) в выбранной упаковке.

> Следующая под-косточка: **упаковка ядра** — выбор как удовлетворение этих требований; затем — **фактическая сила конкретных адаптеров** (наложение фактического покрытия → «защищено / незащищено»).
