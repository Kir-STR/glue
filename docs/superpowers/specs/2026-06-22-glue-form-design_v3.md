# Glue — форма продукта (design, v3)

**Дата:** 2026-06-22
**Статус:** рабочий design формы. **Заменяет `2026-06-22-glue-form-design_v2.md`** (ломающая переработка модели внешней границы — §7/§8: «CLI — единственная внешняя машинная граница» уточнено до «единственная *публичная интеграционная* точка» после появления живого браузерного клиента; `v2` остаётся историей, помечен как заменённый). Ведётся поверх застывшей концепции (`2026-06-22-glue-concept-design.md`); концепцию не переписывает.
**Скоуп:** форма продукта Glue — чем Glue является в момент исполнения, упаковка, клиенты, человеческое представление карты, слой доставки. Концептуальная модель (линзы, рёбра, судьи, петля промоции) — в концепт-спеце.
**Происхождение:** раунд формы. `v1` исходил из модели «защита от обхода» (TCB / актор-угроза / break-glass); `v2` сузил до «помогаем избегать честных ошибок, не защищаем от намеренного обхода». `v3` закрывает под-косточку упаковки: человеческое представление карты (визуализация в базовой поставке) + связь со `invoker` (доставка). Метод — проза, по одному решению, взаимная адъюдикация.

---

## 1. Корневая формула — «помогаем избегать ошибок»

**Целевой актор сотрудничает.** Glue помогает кооперативному агенту (и человеку) **не ошибаться**: не забыть правило, не потерять контекст, не пойти не туда. **Намеренный обход — вне модели угроз**: это твой проект, ты его хозяин и можешь сделать с ним что угодно; строить оборону против самого себя бессмысленно.

Отсюда — что Glue делает и чего не делает:

- **делает:** ловит честные ошибки в момент действия + **честно диагностирует непроверенное** («тут я не проверил»);
- **не делает:** не препятствует тому, кто *хочет* обойти; локальные проверки — **guardrails (ремень безопасности + пищалка), не граница безопасности**.

**Гибрид упаковки выводится из этой модели** (не выбирается из меню):

- **локальный лёгкий инструмент** — проверки в момент работы (агент дёргает через хук); ловит ошибки, обходимость не важна;
- **платформенный контур** (GitHub / CI / branch protection) — для **наружных действий** (push / merge / deploy): даёт более сильную защиту от **случайного** обхода и существует уже сейчас.

**Источник правды — файлы.** Состояние инструмента производно и пересобираемо; второй авторитетной БД нет (компиляция файлов → валидированный снимок + индекс + last-known-good).

**Сердцевина проверок** (бывшие «семь обещаний ядра») сохраняется как машинная — но ради **надёжной помощи**, не неуязвимости: валидация набора и атомарная активация; разрешение назначений; фиксация снимка evidence; запуск чекеров; агрегация + fail-closed для блокирующих; машинная конфликт-детекция; трасса и last-known-good.

## 2. Что должно быть исправно, чтобы помощник не врал

(Переобоснование бывшей «границы доверия / TCB». Теперь это **не** граница против злоумышленника, а перечень того, что должно **работать корректно**, иначе сигнал помощника станет ложным.)

- **Предсказуемость (детерминизм):** одинаковая ситуация → одинаковый ответ. Помощник, который «то ловит, то нет», бесполезен. Вход = канонический снимок; время/сеть/случайность карантинятся в фазе сбора (концепт-спец §5).
- **Честность непокрытого:** если проверка не сработала или класс действия не покрыт — сказать прямо, **не** создавать иллюзию контроля. Худшее для помощника — ложное спокойствие. (Бывшее «незащищено = диагностируется» остаётся, и здесь оно центральное.)
- **Сборщик собирает честно ↔ реальность недоверена:** снимок evidence обеспечивает воспроизводимость (помощник показывает ровно то, что видел сборщик). Это нужно для предсказуемости и честности, **не** как анти-подделка. Сырой вход не «доверен/недоверен» в смысле безопасности — он просто фиксируется снимком, чтобы ответ был воспроизводим.
- **Останов при поломке проверки на опасном (fail-closed для блокирующих):** если чекер упал — лучше притормозить и сказать «не смог проверить», чем молча пропустить и внушить, что всё ок. last-known-good: невалидный набор не применяется, продолжает действовать предыдущий валидный.

Криптографическая неподделываемость, изоляция под отдельными правами, «компонент, до которого не дотянется злоумышленник» — **не входят** в базовый режим (см. §6).

## 3. Карта проверок — что когда проверяется

(Бывшая «логическая карта доверия». Механика композиции сохраняется; язык обороны снят.)

Действие набирает проверки от своих признаков; **все применимые проверки должны пройти** (конъюнкция — не теряем ни одной, иначе пропустим ошибку). Два рода машинной проверки:

- **блокирующая (enforcing-M)** — fail-closed, останавливает действие;
- **сигнальная (advisory-M)** — мягкая, деградирует, подсказывает, **не** останавливает.

| Признак действия | Что добавляется |
|---|---|
| ø (обычная нечувствительная правка) | ничего; свободная работа |
| чтение / навигация / карта / анализ | ничего (by design) |
| под ограничением | блокирующая проверка (рантайм-судья) |
| наружу (outward-facing: публичный PR, публикация, деплой) | **operator-gate** — отдельный гейт, буквальное «да» |
| разрушительное / высокорисковое (force-push, reset --hard, удаление) | **operator-gate** — отдельный гейт каждый раз, форма **report+STOP** |
| меняет несущее звено модели | блокирующая проверка + осознанное подтверждение по типу перехода (§3.1) |
| промоция → Правило | подтверждение нормы; структурная проверка — сигнальная |
| промоция → Ограничение | разворачивается в «меняет несущую модель» |

**`operator-gate` и audit-trail сохраняются — но как дисциплина осознанного действия, не защита от злоумышленника.** Они помогают человеку *не сделать наугад* то, что заслуживает паузы; их назначение — осознанность, а не неуязвимость.

### 3.1. Изменение несущей модели — осознанные фазы

**Критерий несущего звена** (не закрытый перечень): несущее — **любое управляемое звено, изменение которого меняет вычисление, доставку или применение вердикта**. Примеры (не исчерпывающие): само ограничение, чекер, источник (provenance), назначение. Критерий согласован с осью «несущести» концепт-спец §7.

```
авторизация перехода → правка → проверка набора
  → нужные подтверждения → атомарная активация
```

Подтверждения задаёт тип перехода (осознанность, не оборона): новое вето — G1 (норма) → G2 (включение); ослабление/снятие — teardown-гейт; техническая замена без смены смысла — простая авторизация правки.

## 4. Где что проверяется

- **Локально, в момент работы — guardrails.** Лёгкий инструмент на машине, агент вызывает через хук. Ловит ошибки агента. Обходимость не важна и не цель.
- **На платформе — для наружных действий.** GitHub / CI / branch protection дают более сильную защиту от **случайного** обхода push / merge / deploy. Эта защита уже существует — Glue на неё опирается, а не дублирует.

Сила гарантии теперь не «независимость от злоумышленника», а **«ловит ли типичную ошибку»** — и для повседневных локальных действий лёгкого guardrail достаточно.

---

## 5. Строгий профиль — отложен

Adversarial / strict profile — защита от **намеренного** обхода (криптографическая подлинность подтверждений, независимый держатель, неподделываемый журнал, политика «нельзя обойти», break-glass) — **явно отложен**. Это другой уровень сложности; на упаковку базового продукта он **не влияет**. Если когда-нибудь понадобится — отдельный раунд поверх базовой формы.

---

## 6. Требования к упаковке (под модель «помощь»)

- **лёгкий локальный инструмент**, вызываемый агентом через хук, для проверок в момент работы;
- **опора на платформенный контур** (GitHub / CI) для наружных действий — не дублировать;
- **файлы — источник правды**, состояние инструмента производно и пересобираемо;
- **детерминизм** проверок (предсказуемый сигнал);
- **честная диагностика непокрытого** (не врать, что проверено);
- **fail-closed для блокирующих** на опасном (останов вместо молчаливого пропуска).

## 7. Упаковка — решение

**Внешняя граница — процессный контракт CLI (вариант A).** Все *публичные* интеграции вызывают Glue как **команду**; никто не обязан импортировать его код. (Уточнение границы при появлении браузерного клиента — §8.6.)

**Физически Glue — три вещи (+ визуальный клиент, §8):**

1. **кит артефактов** — распространяемый набор правил / навыков / шаблонов / знаний (для Claude Code, Codex, Gemini);
2. **`glue`** — исполняемая команда: кодовые судьи + запросы к карте. Внутри может быть кодовая библиотека ради тестируемости и переиспользования, но это **внутренняя архитектура, не публичный способ интеграции**;
3. **адаптеры** — тонкая привязка событий конкретного агента или CI к команде (наш «тупой адаптер» — без нормативной логики).

Продукт гибридный по составу, но имеет **одну публичную интеграционную границу — контракт CLI**.

**Контракт (структурированный, версированный):**

- событие → **JSON в stdin**;
- вердикт + трасса → **JSON в stdout** (`--format=json`);
- человекочитаемое представление того же ответа → **stdout** (`--format=human`);
- диагностический текст → **отдельно (stderr)**; stderr — **только** диагностика, не ответ;
- итог дополнительно закреплён **кодом выхода**;
- интерактивный текстовый вывод — **представление того же ответа**, не отдельный путь.

Claude Code, Codex, Gemini и CI используют **один** контракт `событие → glue → структурированный вердикт`; различаются **только адаптеры**. (Согласовано с концепт-спец §6: агент — запросный движок со структурированным ответом; человеческое представление — визуальный клиент того же движка.)

**Доставка и онбординг — отдельный слой (§9).** Первичная дверь — **плагин Claude Code** (как `invoker`): разворачивает кит артефактов в проект, ставит/регистрирует команду `glue`, прописывает хуки-адаптеры, преднастраивает старт. Инвариант:

> Плагин **разворачивает** ядро (кит + CLI + адаптеры + визуальный клиент), но **не является** им и **не запирает** его.

Команда `glue` и кит — самостоятельны. Для Claude Code установка идёт через плагин; для Codex / Gemini — свой установщик позже, **рантайм тот же** (та же команда, тот же контракт, свой тонкий адаптер). Glue так догфудит себя — он сам поставлен как кит артефактов.

## 8. Человеческое представление карты — визуализация в базовой поставке

Концепт-спец §6 задал модель: одна каноническая графовая модель → два потребителя (агент = запросный движок; человек = навигационное представление, read-only к модели, «read-only ≠ статичная картинка»). Этот раздел фиксирует **форму** человеческого представления.

### 8.1. Визуализация входит в базовую поставку

Визуальное представление карты — **ключевой смысл продукта**, не вспомогательный слой к агентскому пути. Текстовый суррогат (ASCII-дерево / таблицы CLI) малоинформативен для навигации по графу (импакт-анализ, трасса происхождения, соседство) и **отвергнут** как «приличная карта». Поэтому интерактивное визуальное полотно входит в **базовую** поставку (в отличие от строгого профиля §5: тот меняет *гарантии* и отложен, визуализация меняет только *представление* и нужна сразу).

Требование — **интерактивный клиент, не статичное изображение** (раскрытие соседей, выбор трассы, фильтры по линзам, переход к источнику). Статичность *доставки* клиента (web-ресурсы могут быть статическими файлами) этому не противоречит — речь о статичности *представления*.

### 8.2. Упаковка — `B`, направление зависимостей — `A`

Развилка «отдельный клиент vs встроено в `glue`» смешивала две независимые оси. Решение разводит их:

- **по упаковке — `B`:** `glue map --serve` запускает локальное полотно; **один установленный продукт, одна команда** — отдельный устанавливаемый web-компонент на старте не нужен;
- **по направлению зависимостей — `A`:** `core/query` и `web-client` — **раздельные модули**; ядро **не импортирует** UI; CLI выступает **composition root** (связывает модули и поднимает сервер).

> **Чекпойнт:** один устанавливаемый продукт и одна команда, но раздельные модули `core/query` и `web-client`.

### 8.3. Архитектурная схема

```
browser UI → session API → query layer → canonical model
```

Сервер **локальный, сессионный, read-only** — мутационных методов у него нет (мутация вынесена за границу карты, концепт-спец §6: gated workflow, не прямое редактирование полотна).

### 8.4. Query layer — единственный traversal; разделение ролей

Вся **семантическая навигация** выполняется в query layer: provenance, impact, соседство, линзы. И CLI-`--format`, и session API — **два транспорта над одним query layer**; UI **не реализует** параллельную графовую логику.

UI вправе выполнять **только представление**: layout, pan/zoom, локальную подсветку.

> **Инвариант полноты** (висит на query layer, не на формате): query layer обязан покрывать **все** навигационные операции человека. Web-клиент **не должен требовать изменений канонической модели или query layer**; если потребует — базовый контракт спроектирован недостаточно полно.

### 8.5. Контракт потоков

- `--format=json` → структурированный ответ в **stdout**;
- `--format=human` → человекочитаемая карта в **stdout**;
- **stderr** → только диагностика.

(Человекочитаемая карта — это **ответ**, не диагностика; поэтому stdout, не stderr. Уточняет прежнюю формулировку §7, где stdout нёс только JSON.)

### 8.6. Граница — переформулирована (ломающая правка к v2 §7)

С появлением живого браузерного клиента «CLI — единственная буквальная внешняя граница» больше неверно. Корректная формулировка:

> **CLI — единственная *публичная интеграционная* точка.** `map --serve` открывает **внутренний сессионный транспорт** только для комплектного UI.

Свойства session API:

- **непубличный, без внешних гарантий совместимости** (в отличие от CLI-контракта, который версионируется — `versioning.md`) — именно отсутствие обещания совместимости делает его внутренним, а не второй публичной границей;
- **но несёт внутренний version/hash handshake:** комплектный UI и сервер обязаны **обнаруживать рассинхрон сборки** — устаревшая вкладка против свежего сервера должна падать явно, а не тихо неверно рендерить (иначе нарушается «честность непокрытого» §2);
- **эфемерный порт + токен сессии** изолируют *сессию*, а не объявляют публичный контракт.

## 9. Связь со `invoker` — доставка kit-слоя

Доставка Glue распадается на два слоя, и `invoker` (file-layer scaffolder: раскладывает выбранные модули правил в `.claude/rules` + `CLAUDE.md`, движки claude/agents/gemini) закрывает только один:

- **kit-слой** (правила / навыки / шаблоны / знания, многодвижковая раскладка) — `invoker` сейчас закрывает доказанную часть: **раскладку встроенных generic-модулей** (по `modules.json` — выбор из встроенного каталога; кастомные Glue-модули — §9.2, непроверено);
- **runtime-слой** (установка/регистрация команды `glue`, проводка хуков-адаптеров, преднастройка старта, визуальный клиент) — `invoker` не умеет и по назначению не должен; это собственный тонкий установщик Glue в любом случае.

### 9.1. `invoker` — pinned build-tool (A на авторинге, B на поставке)

Развилка «runtime-зависимость vs переписать механику» смешивала build-time и runtime. Решение:

- на этапе **авторинга/сборки** kit-слоя Glue использует **зафиксированную (pinned) версию** `invoker` для той части кита, которую он **фактически поддерживает**;
- **release** содержит уже подготовленные **многодвижковые артефакты** + **manifest происхождения** (какой инструмент и какая версия произвели каждый артефакт);
- **установщик Glue** раскладывает этот готовый кит сам;
- пользователю и runtime Glue `invoker` **не требуется**.

> **Чекпойнт:** `invoker` — pinned **build-tool**, не runtime-зависимость и не публичный контракт Glue. Контроль дрейфа: изменение генерации проявляется как **проверяемый diff release-артефактов**.

Так уже решённая многодвижковая раскладка не переписывается, но формат и темп эволюции `invoker` **не попадают в runtime-критический путь** Glue.

### 9.2. Граничный случай — кастомные модули (дизайн держится при любом ответе)

`invoker` (по `modules.json`) раскладывает выбор из **встроенного** каталога. Если кастомные Glue-специфичные модули (правила команды `glue`, навык карты, шаблоны) `invoker` на вход **не принимает** — непокрытая Glue-специфичная часть кита генерируется **Glue-native инструментом**. Manifest фиксирует производителя **каждого** артефакта (invoker / Glue-native).

> Так проверка возможности (`принимает ли invoker кастомные модули`) — **verification к этапу реализации**, а не опровержение дизайна: дизайн корректен при любом ответе, меняется лишь широта покрытия build-tool.

## 10. Что остаётся открытым

- **Строгий профиль** (§5) — отложен осознанно; отдельный раунд при необходимости.
- **Содержание контракта CLI** — **семейство схем под общей версионированной оболочкой** (`versioning.md`: версия в имени файла). Контракт несёт **разные операции одной публичной границы**: проверки (`событие → вердикт/трасса`, stdin→stdout) и запросы карты (`запрос → graph result`, тот же query layer §8.4). Конкретные схемы каждой операции — уровень спеки/реализации, не формы.
- **Verification к реализации** (§9.2): принимает ли pinned `invoker` кастомные модули — определяет широту покрытия build-tool.

Базовая **форма** продукта этим определена: гибрид (кит + `glue` CLI + адаптеры + визуальный клиент), одна публичная интеграционная граница (контракт CLI), визуализация карты в базовой поставке как клиент того же query layer, доставка через плагин Claude Code с `invoker` как pinned build-tool kit-слоя.
