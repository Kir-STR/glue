## 1. Что такое Glue

**Glue — подход к работе с проектными артефактами и набор плагинов, который его воплощает.** Он не управляет кодом, не заменяет инструменты разработки / планирования / таск-трекинга и не оркеструет агентов. Glue хранит, связывает и показывает артефакты вокруг работы — оставляя саму работу (и разработку, и инфраструктуру) свободной.

Цель — сделать структуру проекта и его ограничения наблюдаемыми, управляемыми и трассируемыми, с приоритетом происхождения (provenance-first).

**Структура первична, судья — поверх.** Главный guardrail Glue — сама связанная структура: артефакты вынесены, адресуемы, видимы, имеют provenance и не тонут в контексте. Кодовый судья — самый жёсткий, но и самый узкий механизм (только формализуемые ограничения), не центр модели. Glue помогает кооперативному агенту не ошибаться по-честному; намеренный обход — вне модели (помощь, не оборона).

**Границы (что Glue НЕ делает):**

- не пишет код и не провиженит/деплоит инфраструктуру;
- не запускает навыки и не оркеструет процессы;
- может **запретить** действие через кодового судью — но запрет ≠ оркестрация.

**Что Glue берёт как есть vs извлекает:**

- **Берёт как есть:** Правила (`CLAUDE.md`, `.claude/rules`, `AGENTS.md`, `GEMINI.md`), Навыки (skills + hooks), Знания (документация — в контекст не пихаем).
- **Извлекает спрятанное:** Решения → ADR; Ограничения → отдельный md + кодовый судья; инфра-знание → спеки + правила работы.
- **Мотивы извлечения:** (1) гигиена контекста; (2) enforceability (нельзя обойти); (3) адресуемость / переиспользование отдельными агентами; (4) трассируемость.

## 2. Хребет модели — Provenance-first

Модель **гибридная с привилегированным хребтом**. Главная цепочка — **Решение → Ограничение → Проверка**, но **происхождение ограничения обязательно**.

- **Авторитетный источник** ⊇ { **ADR** (предпочтительный *внутренний* случай), внешнее требование / политика, унаследованное правило с зафиксированным происхождением }.
- **Инвариант:** всякое вето замыкается на авторитетном источнике (не обязательно ADR). Поэтому принцип — **provenance-first**, где ADR — частный (предпочтительный) случай.
- Ограничение **без источника** допустимо **только как временное состояние** с явным статусом и диагностикой.

**Полная трасса вето:** `итог гейта → результаты проверок → прогон → назначение → (чекер + ограничение) → источник`.

## 3. Онтология: липкие артефакты

**Род — «липкий артефакт»** (совокупность — «липкие артефакты проекта»). Имя завязано на бренд Glue: клей держит «приклеенное» рядом с работой и связывает между собой.

«Предписание» — *семантический подтип*: часть артефактов предписывает «как делать», часть только описывает факты (Знания).

### Пять видов артефактов

| Вид | Источник | Форма | Контроль |
|---|---|---|---|
| **Знание** | внешний | справочный корпус (llms.txt / документация); в контекст не грузим | — |
| **Правило** | внутренний | проза-директива (`CLAUDE.md` / `.claude/rules` / `AGENTS.md` / `GEMINI.md`) | требует обдумывания агентом |
| **Навык** | внутренний | процедура (skill, вызывается командой) | можно вызвать и получить результат |
| **Решение** | внутренний | выбор + обоснование (ADR) | LLM-судья (соответствие) |
| **Ограничение** | внутренний | машиночитаемый предикат | кодовый судья, **вето** |

- **Правило** (ранее «Инструкция») — имя согласовано с `.claude/rules`. Не конфликтует с «Ограничением»: мягкая проза ↔ жёсткий предикат.
- **Ограничение** становится ограничением **только когда кодовая проверка реализована и подключена** (формализуемость ≠ вето).
- **Не форсируем** единый тип узла / единое хранение: у ADR / навыка / документа / предиката могут быть разные схемы (уровень реализации).

### Три оси, режущие поперёк видов

Виды — это «что за артефакт». Поперёк них идут три ортогональные оси (не отдельные виды):

- **`env-scope` — ось вердикта** (*где*: local / staging / production). Одно ограничение блокирует на production, предупреждает на local. Ось едет с движком судьи — везде, где есть ограничения. `env-scope` якорится **декларативным профилем** окружения (дёшево, детерминированно); живое состояние НЕ определяет активный профиль — оно снимается снимком (§5) и служит доказательством для **опциональных** инфра-чекеров «реальность ↔ декларация».
- **Предмет — ось dev / infra** (*про что*). Пять видов применяются равноправно к разработке и инфраструктуре; инфра — первоклассный предмет, не довесок. Один артефакт может относиться к обоим (правило миграции связывает код, базу и production). Композируется с `env-scope`.
- **Мягко / жёстко — ось контроля** (*как*): интерпретация агентом → LLM-судья → кодовый судья (§4).

Имена видов/рода — нейминг **содержания**. Имена-идентификаторы (slug'и, ключи YAML/JSON, импорты) — ASCII, отдельная политика (`glossary.md`).

## 4. Спектр судей (мягко ↔ жёстко)

**Спектр контроля — не бинар:** интерпретация агентом (мягко) → LLM-судья → кодовый судья (жёстко). Один артефакт может трогаться разными режимами: тело навыка читает агент, а *запуск* гейтит кодовый судья.

**Судья — роль:** LLM-судья (мягко, рекомендательно, без вето) ↔ кодовый судья (жёстко, вето). Граница = **формализуемость**. Повторяющийся LLM-флаг → кандидат в ограничение (§9).

## 5. Судьи и проверки

### Сущности

*Статические (декларативные):*

- **Ограничение** — машиночитаемый предикат-спека + provenance + `severity`.
- **Чекер (реализация)** — детерминированный код, вычисляющий предикат (возможно параметризуемый).
- **Гейт** — точка перехвата (pre-skill-launch / pre-merge / pre-action …).
- **Назначение** — связка `ограничение → чекер на гейте`, с `env-scope` и версией. Одно ограничение обычно считается одним чекером; несколько гейтов/окружений выражаются несколькими назначениями (без отдельной M:N-машинерии).

*Рантайм (события):*

- **Прогон гейта** — несёт снимок входа.
- **Результат проверки** — `pass | fail | error`.
- **Итог гейта** — агрегат allow/deny.

### Выражение предиката

- Именованный **версионированный чекер** + **строго валидируемый frontmatter** (`checker-id + параметры + env-scope + режим применения`).
- **md** = человеческое описание + provenance (для человека и LLM-судьи).
- Неизвестный чекер / невалидные параметры / несовместимая версия = **ошибка конфигурации**, а не молчаливый пропуск.
- **Markdown — спецификация только при машиночитаемой структуре.** Свободный текст не исполняется детерминированно.

### Детерминизм — три фазы прогона

1. **Сбор доказательств** (грязная фаза, карантин недетерминизма): intent, нормализация аргументов, снимок окружения, живое состояние читается **один раз и замораживается** как evidence.
2. **Оценка** (чистая): чекеры над `(версия чекера, версия ограничения, канонический снимок)` → результат.
3. **Агрегация** (чистая): результаты → итог гейта.

- **Вход** = `intent + нормализованные аргументы + снимок окружения + снимок доказательств + версия ограничения + версия чекера`.
- Время / сеть / случайность — **запрещены или передаются версионированным входом**.
- Одинаковые версии чекера и ограничения + одинаковый канонический снимок → одинаковый результат (включая `error`).

### Агрегация

- **severity** — серьёзность нарушения; часть спеки ограничения (изменение `severity` = новая версия ограничения).
- **enforcement** — `block` / `warn`, env-зависимый; понижение `block→warn` — явное, с provenance.
- **result state** — `pass` / `fail` / `error`.
- Стратегия — **deny-overrides**. Авто-разрешение нормативных конфликтов не делается (конфликты — §10).

### Алгоритм прогона гейта

1. Отобрать применимые назначения по `гейт × env-scope × intent`.
2. По каждому: снимок → чекер → результат `{pass, fail, error}`.
3. Свёртка `result × enforcement`:
   - `pass` → ок;
   - `fail + block` → блокирующее нарушение;
   - `fail + warn` → предупреждение;
   - `error + block` → **fail-closed** → блокирующее нарушение (причина «checker error»);
   - `error + warn` → предупреждение (fail-open только для warn).
4. **Итог = deny, если есть хоть одно блокирующее нарушение**, иначе allow. К отчёту — все нарушения + предупреждения по severity.

### Провенанс ошибки

- **config error** (невалидный frontmatter) → ловится при валидации конфигурации, набор не грузится.
- **collection failure** (не снят evidence) → рантайм, fail-closed для блокирующих.
- **evaluation error** (чекер падает на валидном снимке) → рантайм, fail-closed для блокирующих + помечается как дефект чекера.

### Операционная безопасность загрузки

- Отказ загружать невалидный набор — **атомарный**, с сохранением **последней валидной конфигурации** (last-known-good). Невалидный набор не применяется; продолжает действовать предыдущий валидный.
- **Нормативные конфликты** выявляются при валидации / LLM-анализе (§10), а не разрешаются «выбором победителя» во время опасного действия.

## 6. Карта: одна модель, два представления

Одна каноническая графовая модель (узлы + рёбра, §3 / §5 / §7) — **два потребителя:**

- **Агент — запросный движок.** Программный обход графа (применимые ограничения по `intent × env-scope`, трасса вето, provenance). Ответ — **структурированный** (подграф / трасса / узлы), не пиксели.
- **Человек — навигационная визуализация.** Аудит, навигация, разбор накопителя, импакт-анализ («что сломается, если убрать ADR» = обход несущих рёбер вверх).

**Решения:**

- **Карта-UI = только человеческое представление**, а не сам Glue. Агентский интерфейс — API обхода той же модели.
- **Один обходной слой, два клиента:** навигационные примитивы человека = те же запросы движка. Визуализация — визуальный клиент того же движка, не параллельный traversal-слой.
- Человек: **read-only к модели, интерактив по навигации** (фильтры по видам, раскрытие соседей, выбор трассы, импакт-анализ, переход к источнику). «Read-only» ≠ статичная картинка.
- **Мутация вынесена за границу карты:** редактирование из карты не включаем. Карта может *инициировать* отдельную команду изменения, но сама мутация проходит обычный контракт + проверку + гейт (§8) — gated workflow, не прямое редактирование полотна.

## 7. Типология рёбер — по несущести

**Главная ось — «несущесть»** (участвует ли ребро в вердикте/трассе). **Класс ребра определяет чувствительность** (§8): несущие — под гейтом, прочие — зеркало.

**Несущие (load-bearing):**

- **`derives_from`** (источник → ограничение) — provenance вето, право блокировать **сейчас**.
- **`назначение` / `enforces`** — связка `ограничение → чекер на гейте` (с `env-scope`, версией); по ней вычисляется вето (§5).
- `applies-in-env` — **не отдельное ребро**; `env-scope` — измерение внутри назначения.

**Справочные (informational):**

- **`references`** / see-also — упоминание без гарантий.
- **`promoted_from`** (lineage) — как артефакт появился (§9).
- **`supersedes`** — новое ограничение пришло на смену старому (§9), новое→старое.

**Структурные (structural):**

- **`contains`** / part-of — файл содержит правило.

**Не сливать три «происхождения»:** `derives_from` (вето сейчас, несущее) ≠ `promoted_from` (как появился, справочное) ≠ `supersedes` (что заменило, справочное).

## 8. Навыки-мутаторы и чувствительные действия

- **Навык = Claude Code skill** (вызывается командой; гейтинг запуска через хуки — уровень реализации).
- **Двойственность навыка:** **артефакт** (тело читает агент) + **действие** (запуск перехватывается гейтом).
- **Owned ↔ observed:**
  - **owned** (свои: `retro`, `save`, структурирующий навык §9) — Glue поставляет, видит на карте, при необходимости мутирует;
  - **observed** (чужие: superpowers и пр.) — Glue видит как артефакты, **read-only**, но **их запуск тоже гейтится**.
- **Ключевой синтез: ограничение гейтит ДЕЙСТВИЕ, не навык.** То же `push` может прийти из чужого навыка, своего или напрямую от агента. Ограничение назначается на **гейт (точку перехвата)**, не на навык.
- **Гибрид «зеркало vs дверь»:**
  - по умолчанию — **зеркало** (изменение видим постфактум, обновляем карту);
  - для **чувствительных** артефактов — **«дверь»** = тот же гейт-механизм на мутацию (отдельной конструкции «двери» нет).
- **Чувствительные артефакты:** **ограничения** (ослабить/снять вето опасно) + **provenance и связи источник→ограничение** (подделка подрывает хребет).
- **НЕ дублируем гейтом:** **правила** — их изменение идёт через существующий PR + оператор (`retro-loop`). Знания/документация и создание ADR — зеркало (но связь «ADR → действующее ограничение» уже чувствительна как provenance).
- **Класс ребра → чувствительность:** несущие рёбра (§7) — под гейтом; справочные и структурные — зеркало.

## 9. Структура документов и петля промоции

### Гранулярность правил

- **Серединка:** **файл = узел; под-узел материализуется лениво** — когда отдельное смысловое правило нуждается в собственной связи, области применения, происхождении или проверке. Раздел Markdown **сам по себе не обязан** быть узлом. Грубые связи — к файлу, точные (provenance, `derives_from`) — к под-узлу. Касается в основном **Правил** и **Знаний**; у Ограничений / Решений / Навыков одна вещь = один файл.
- **Адресуемость под-узлов опирается на контролируемую структуру**, которую держит **owned структурирующий навык** (умная реорганизация правил/знаний: разложить, сгруппировать, проставить якоря).
- **Три ограждения:**
  1. **Навык предлагает, не молча переписывает** — мутация правил через PR + оператор; Glue не хозяин текста.
  2. **Структура — разметка ради адресности, не переписывание смысла** (стабильные якоря/ID; нюанс прозы сохраняется).
  3. **Опираемся только при валидации** — структура поехала → мягкая деградация к файл-гранулярности **для навигации** (справочные/структурные ссылки), не падение. Но **несущая ссылка** (`derives_from` на под-узел, §7) к неразрешимому точному источнику — это **ошибка конфигурации + last-known-good** (§5), а не молчаливое расширение до файла: иначе провенанс вето тихо огрубляется и вето теряет легитимность.
- Постройка самого навыка — реализация/роадмап; здесь фиксируем только принцип.

### Петля промоции

**Один источник, общий с retro-loop:** сигнал рождается в **разборе только что сделанной работы** + боковой вход «оператор напрямую». Различие петель — не на входе, а **на выходе** (природа вывода и его потребитель).

**Развилка на выходе:**

- «надо помнить / обдумывать» → **правило** (мягкий мир) — короткий хвост;
- «надо механически не пускать» → **ограничение** (жёсткий мир) — длинный хвост;
- «зафиксировать выбор с обоснованием» → ADR/Решение (LLM-судья).

**Два темпа (расцеплены):** часто (конец сессии) агент накидывает предложения в накопитель с предполагаемым типом; реже (отдельно) оператор разбирает.

**Накопители раздельные по типу** (правила / ограничения) — разные мерки на разборе.

**Трёхуровневая организация на разборе:** стопка (тип) → группа ценности (важное / в бэклог / отвергнуть) → сортировка по повторениям. **Повторения (n-count) — вес внутри группы, НЕ автозапуск и НЕ привратник.**

**Хвост ограничений — два независимых операторских «да»** (Gate 1 / Gate 2 split, `operator-gate`):

- **Gate 1 — принять норму** («это правомерно контролировать»). Ничего не блокирует. → между гейтами: реализация чекера + перепроверки.
- **Gate 2 — включить вето** («чекер реализован, корректность доказана, подключаем»). Деструктивный/высокорисковый класс.
- **Gate 2 не наследуется из Gate 1.** Между гейтами реализация может вскрыть неформализуемость → миграция в правило, G2 не наступает.

**Статусная модель кандидата:** `предложен → принят (G1) → реализован → включён (G2) = Ограничение`; боковые исходы `отклонён` / `ушёл в правило`. **Путь кандидата не расширяем** (YAGNI).

**Жизненный цикл живого ограничения** (отдельно, под teardown-гейтом): **работает / приостановлено / снято**. «Заменено» — **не состояние, а ребро `supersedes`** (§7).

**Cleanup после промоции:** кандидат **исчезает из рабочего накопителя**, но остаётся **минимальный неизменяемый след** (id, снимок содержания, источник, итог) как разрешаемая цель `promoted_from` — иначе оборванное происхождение. **Очистка — только после** создания артефакта + записи происхождения + проверки результата.

**Связь с retro-loop:** общий рисунок, раздельные контуры; миграция сигнала между контурами **только через оператора**; обе промоции — через PR.

## 10. Конфликт-детекция

**Принцип:** нормативный конфликт **выявляется заранее и показывается оператору**, не разрешается «выбором победителя» на лету.

**Два слоя:**

- **Машинный** — два ограничения с чёткими предикатами противоречат в одной ситуации / `env-scope`. Ловится при валидации/загрузке набора, детерминированно. **Блокирует активацию нового набора** (Glue продолжает на last-known-good). Стартовый статус — **подтверждён**.
- **Смысловой** — две прозы-правила противоречат только по смыслу. LLM-анализ помечает как **подозрение**, не факт и не вето. Оператор подтверждает/отклоняет.

Конфликт **показывается оператору** с областью пересечения (`гейт × env-scope × intent`) и статусом (`подозрение → подтверждён → разрешён / отклонён`); на карте — компактное ребро **«спорит с»**.

- **Явно заданное исключение / явное решение о старшинстве — НЕ конфликт.** Override легитимен, если у него есть свой артефакт-решение с provenance. Конфликт = противоречие **без** объясняющего артефакта.
- **deny-overrides — аварийное поведение исполнения, не закрытие конфликта.** Разрешение — **только** изменением артефактов либо явным решением оператора; авто-победитель запрещён.

## 11. Связь с правилами проекта

- **`operator-gate`:** Gate 1 / Gate 2 split (петля промоции, мутация чувствительного); `block→warn` = эшелонированная защита (обход не каскадируется).
- **`versioning`:** версии чекера/ограничения; **изменение `severity` входит в версию ограничения**.
- **`retro-loop`:** петля «LLM-флаг → кандидат → ограничение»; общий рисунок эволюции, раздельные контуры.
