# Glue — форма продукта (design, v4)

**Дата:** 2026-06-23
**Статус:** рабочий design формы. **Заменяет `2026-06-22-glue-form-design_v3.md`** (переработка **формы доставки**: монолитный плагин → три самодостаточных плагина, работающих в любом составе; `v3` остаётся историей, помечен как заменённый). Ведётся поверх застывшей концепции (`2026-06-22-glue-concept-design.md`); концепцию не переписывает.
**Скоуп:** форма продукта Glue — упаковка, клиенты, человеческое представление карты, **декомпозиция доставки**. Рантайм-форма (контракт CLI, карта/query layer, визуализация, invoker как pinned build-tool) переносится из `v3` без изменений; меняется только единица поставки.
**Происхождение:** раунд формы. `v1` исходил из модели «защита от обхода»; `v2` сузил до «помогаем избегать честных ошибок»; `v3` закрыл упаковку (контракт CLI + визуализация карты + invoker). `v4` фиксирует **форму доставки**.

Нарезка на три плагина следует **порядку, в котором приходит боль** у растущего проекта, а не типу артефакта как самоцели: (1) правила тонут, нужна прополка и видимость → **П1**; (2) решения нужно фиксировать и контролировать исполнение → **П2**; (3) появляется эксплуатация со своей спецификой → **П3**. Каждая следующая волна накрывает меньше проектов. «Любой состав» требуется потому, что проекты реально стоят на разных ступенях этой лесенки и докупают следующий плагин по мере боли, — **не** ради модульности как самоцели.

**Процесс-инвариант:** форма не переписывает концепцию. Расхождения с концепт-спецем фиксируются замечанием здесь (см. §10.3, §11); концепт-спец и `IDEA.md` не правятся.

**Метод:** проза, по одному решению, взаимная адъюдикация.

## 1. Корневая формула — «помогаем избегать ошибок»

**Целевой актор сотрудничает.** Glue помогает кооперативному агенту (и человеку) **не ошибаться**: не забыть правило, не потерять контекст, не пойти не туда. **Намеренный обход — вне модели угроз**: это твой проект, ты его хозяин и можешь сделать с ним что угодно; строить оборону против самого себя бессмысленно.

Отсюда — что Glue делает и чего не делает:

- **делает:** ловит честные ошибки в момент действия + **честно диагностирует непроверенное** («тут я не проверил»);
- **не делает:** не препятствует тому, кто *хочет* обойти; локальные проверки — **guardrails (ремень безопасности + пищалка), не граница безопасности**.

**Гибрид упаковки выводится из этой модели** (не выбирается из меню):

- **локальный лёгкий инструмент** — проверки в момент работы (агент дёргает через хук); ловит ошибки, обходимость не важна;
- **платформенный контур** (GitHub / CI / branch protection) — для **наружных действий** (push / merge / deploy): даёт более сильную защиту от **случайного** обхода и существует уже сейчас.

**Источник правды — файлы.** Состояние инструмента производно и пересобираемо; второй авторитетной БД нет (компиляция файлов → валидированный снимок + индекс + last-known-good).

**Сердцевина проверок** (бывшие «семь обещаний ядра») сохраняется как машинная — но ради **надёжной помощи**, не неуязвимости: валидация набора и атомарная активация; разрешение назначений; фиксация снимка evidence; запуск чекеров; агрегация + fail-closed для блокирующих; машинная конфликт-детекция; трасса и last-known-good.

## 2. Что должно быть исправно, чтобы помощник не врал

(Переобоснование бывшей «границы доверия / TCB». Теперь это **не** граница против злоумышленника, а перечень того, что должно **работать корректно**, иначе сигнал помощника станет ложным.)

- **Предсказуемость (детерминизм):** одинаковая ситуация → одинаковый ответ. Помощник, который «то ловит, то нет», бесполезен. Вход = канонический снимок; время/сеть/случайность карантинятся в фазе сбора (концепт-спец §5).
- **Честность непокрытого:** если проверка не сработала или класс действия не покрыт — сказать прямо, **не** создавать иллюзию контроля. Худшее для помощника — ложное спокойствие.
- **Сборщик собирает честно ↔ реальность недоверена:** снимок evidence обеспечивает воспроизводимость (помощник показывает ровно то, что видел сборщик). Это нужно для предсказуемости и честности, **не** как анти-подделка.
- **Останов при поломке проверки на опасном (fail-closed для блокирующих):** если чекер упал — лучше притормозить и сказать «не смог проверить», чем молча пропустить и внушить, что всё ок. last-known-good: невалидный набор не применяется, продолжает действовать предыдущий валидный.

Криптографическая неподделываемость, изоляция под отдельными правами, «компонент, до которого не дотянется злоумышленник» — **не входят** в базовый режим (см. §6).

## 3. Карта проверок — что когда проверяется

(Бывшая «логическая карта доверия». Механика композиции сохраняется; язык обороны снят.)

Действие набирает проверки от своих признаков; **все применимые проверки должны пройти** (конъюнкция — не теряем ни одной, иначе пропустим ошибку). Два рода машинной проверки:

- **блокирующая (enforcing-M)** — fail-closed, останавливает действие;
- **сигнальная (advisory-M)** — мягкая, деградирует, подсказывает, **не** останавливает.

| Признак действия | Что добавляется |
|---|---|
| ø (обычная нечувствительная правка) | ничего; свободная работа |
| чтение / навигация / карта / анализ | ничего (by design) |
| под ограничением | блокирующая проверка (рантайм-судья) |
| наружу (outward-facing: публичный PR, публикация, деплой) | **operator-gate** — отдельный гейт, буквальное «да» |
| разрушительное / высокорисковое (force-push, reset --hard, удаление) | **operator-gate** — отдельный гейт каждый раз, форма **report+STOP** |
| меняет несущее звено модели | блокирующая проверка + осознанное подтверждение по типу перехода (§3.1) |
| промоция → Правило | подтверждение нормы; структурная проверка — сигнальная |
| промоция → Ограничение | разворачивается в «меняет несущую модель» |

**`operator-gate` и audit-trail сохраняются — но как дисциплина осознанного действия, не защита от злоумышленника.** Они помогают человеку *не сделать наугад* то, что заслуживает паузы; их назначение — осознанность, а не неуязвимость.

### 3.1. Изменение несущей модели — осознанные фазы

**Критерий несущего звена** (не закрытый перечень): несущее — **любое управляемое звено, изменение которого меняет вычисление, доставку или применение вердикта**. Примеры (не исчерпывающие): само ограничение, чекер, источник (provenance), назначение. Критерий согласован с осью «несущести» концепт-спец §7.

```
авторизация перехода → правка → проверка набора
  → нужные подтверждения → атомарная активация
```

Подтверждения задаёт тип перехода (осознанность, не оборона): новое вето — G1 (норма) → G2 (включение); ослабление/снятие — teardown-гейт; техническая замена без смены смысла — простая авторизация правки.

## 4. Где что проверяется

- **Локально, в момент работы — guardrails.** Лёгкий инструмент на машине, агент вызывает через хук. Ловит ошибки агента. Обходимость не важна и не цель.
- **На платформе — для наружных действий.** GitHub / CI / branch protection дают более сильную защиту от **случайного** обхода push / merge / deploy. Эта защита уже существует — Glue на неё опирается, а не дублирует.

Сила гарантии теперь не «независимость от злоумышленника», а **«ловит ли типичную ошибку»** — и для повседневных локальных действий лёгкого guardrail достаточно.

---

## 5. Строгий профиль — отложен

Adversarial / strict profile — защита от **намеренного** обхода (криптографическая подлинность подтверждений, независимый держатель, неподделываемый журнал, политика «нельзя обойти», break-glass) — **явно отложен**. Это другой уровень сложности; на упаковку базового продукта он **не влияет**. Если когда-нибудь понадобится — отдельный раунд поверх базовой формы.

---

## 6. Требования к упаковке (под модель «помощь»)

- **лёгкий локальный инструмент**, вызываемый агентом через хук, для проверок в момент работы;
- **опора на платформенный контур** (GitHub / CI) для наружных действий — не дублировать;
- **файлы — источник правды**, состояние инструмента производно и пересобираемо;
- **детерминизм** проверок (предсказуемый сигнал);
- **честная диагностика непокрытого** (не врать, что проверено);
- **fail-closed для блокирующих** на опасном (останов вместо молчаливого пропуска).

## 7. Рантайм-форма — контракт CLI (перенос из v3, без изменений)

**Внешняя граница — процессный контракт CLI (вариант A).** Все *публичные* интеграции вызывают Glue как **команду**; никто не обязан импортировать его код. (Уточнение границы при появлении браузерного клиента — §8.6.)

**Физически рантайм Glue — три вещи (+ визуальный клиент, §8):**

1. **кит артефактов** — распространяемый набор правил / навыков / шаблонов / знаний (для Claude Code, Codex, Gemini);
2. **`glue`** — исполняемая команда: кодовые судьи + запросы к карте. Внутри может быть кодовая библиотека ради тестируемости и переиспользования, но это **внутренняя архитектура, не публичный способ интеграции**;
3. **адаптеры** — тонкая привязка событий конкретного агента или CI к команде («тупой адаптер» — без нормативной логики).

Продукт гибридный по составу, но имеет **одну публичную интеграционную границу — контракт CLI**. **Декомпозиция доставки (§10) этого не меняет:** при любом составе плагинов рантайм схлопывается к одному `glue`, одной карте, одному контракту.

**Контракт (структурированный, версированный):**

- событие → **JSON в stdin**;
- вердикт + трасса → **JSON в stdout** (`--format=json`);
- человекочитаемое представление того же ответа → **stdout** (`--format=human`);
- диагностический текст → **отдельно (stderr)**; stderr — **только** диагностика, не ответ;
- итог дополнительно закреплён **кодом выхода**;
- интерактивный текстовый вывод — **представление того же ответа**, не отдельный путь.

Claude Code, Codex, Gemini и CI используют **один** контракт `событие → glue → структурированный вердикт`; различаются **только адаптеры**.

> **Доставка и онбординг — отдельный слой (§9, §10).** В `v3` первичной дверью был **один** плагин Claude Code. `v4` заменяет это на **три** плагина (§10); инвариант сохраняется: плагин(ы) **разворачивают** ядро (кит + CLI + адаптеры + визуальный клиент), но **не являются** им и **не запирают** его. Команда `glue` и кит самостоятельны.

## 8. Человеческое представление карты — визуализация в базовой поставке (перенос из v3)

Концепт-спец §6 задал модель: одна каноническая графовая модель → два потребителя (агент = запросный движок; человек = навигационное представление, read-only к модели, «read-only ≠ статичная картинка»).

### 8.1. Визуализация входит в базовую поставку

Визуальное представление карты — **ключевой смысл продукта**, не вспомогательный слой. Текстовый суррогат (ASCII-дерево / таблицы CLI) малоинформативен для навигации по графу и **отвергнут** как «приличная карта». Интерактивное визуальное полотно входит в **базовую** поставку.

Требование — **интерактивный клиент, не статичное изображение** (раскрытие соседей, выбор трассы, фильтры по линзам, переход к источнику). Статичность *доставки* клиента (web-ресурсы могут быть статическими файлами) этому не противоречит — речь о статичности *представления*.

### 8.2. Упаковка — `B`, направление зависимостей — `A`

- **по упаковке — `B`:** `glue map --serve` запускает локальное полотно; **один установленный продукт, одна команда**;
- **по направлению зависимостей — `A`:** `core/query` и `web-client` — **раздельные модули**; ядро **не импортирует** UI; CLI выступает **composition root**.

### 8.3. Архитектурная схема

```
browser UI → session API → query layer → canonical model
```

Сервер **локальный, сессионный, read-only** — мутационных методов у него нет (мутация вынесена за границу карты, концепт §6: gated workflow).

### 8.4. Query layer — единственный traversal; разделение ролей

Вся **семантическая навигация** выполняется в query layer: provenance, impact, соседство, линзы. И CLI-`--format`, и session API — **два транспорта над одним query layer**; UI **не реализует** параллельную графовую логику. UI вправе выполнять **только представление**: layout, pan/zoom, локальную подсветку.

> **Инвариант полноты** (висит на query layer, не на формате): query layer обязан покрывать **все** навигационные операции человека. Web-клиент **не должен требовать изменений канонической модели или query layer**.

> **Карта/query/визуализация — общий КОДОВЫЙ компонент (§10.2).** «П1 без код-судьи» (§10.1) **не** значит «без кода вообще»: карта, query layer и визуализация — общая кодовая часть, едут с каждым плагином и дедуплицируются. Код-судьи (движок вето) нет только в П1; карта-код есть везде. При составе из одного П1 query layer **чисто деградирует** к графу без узлов-ограничений (нет вето → нет вето-узлов), а не повисает на отсутствующих узлах.

### 8.5. Контракт потоков

- `--format=json` → структурированный ответ в **stdout**;
- `--format=human` → человекочитаемая карта в **stdout**;
- **stderr** → только диагностика.

### 8.6. Граница

> **CLI — единственная *публичная интеграционная* точка.** `map --serve` открывает **внутренний сессионный транспорт** только для комплектного UI.

Свойства session API:

- **непубличный, без внешних гарантий совместимости** (CLI-контракт — версионируется, `versioning.md`);
- **внутренний version/hash handshake:** комплектный UI и сервер обязаны **обнаруживать рассинхрон сборки** — устаревшая вкладка против свежего сервера падает явно, а не тихо неверно рендерит (иначе нарушается «честность непокрытого» §2);
- **эфемерный порт + токен сессии** изолируют *сессию*, а не объявляют публичный контракт.

> **Важно (см. §10.2):** handshake §8.6 ловит **build skew** (равенство хеша UI↔сервер), но **не** ловит **семантическую несовместимость** дедуплицированных компонентов (новый движок прогоняет чекер, написанный под старый контракт). Для дедупа нужна отдельная проверка совместимости — §10.2.

## 9. invoker — pinned build-tool kit-слоя (перенос из v3)

Доставка распадается на kit-слой (правила/навыки/шаблоны/знания, многодвижковая раскладка) и runtime-слой (установка `glue`, проводка адаптеров, преднастройка, визуальный клиент). `invoker` (file-layer scaffolder) закрывает **kit-слой**; runtime-слой — собственный установщик Glue.

### 9.1. Контракт build-tool

- на этапе **авторинга/сборки** kit-слоя Glue использует **зафиксированную (pinned) версию** `invoker` для той части кита, которую он **фактически поддерживает** (доказана раскладка встроенных generic-модулей по `modules.json`);
- **release** содержит уже подготовленные **многодвижковые артефакты** + **manifest происхождения** (какой инструмент и какая версия произвели каждый артефакт);
- **установщик Glue** раскладывает готовый кит сам; пользователю и runtime `invoker` **не требуется**.

> **Чекпойнт:** `invoker` — pinned **build-tool**, не runtime-зависимость и не публичный контракт Glue. Контроль дрейфа: изменение генерации — **проверяемый diff release-артефактов**.

### 9.2. Граничный случай — кастомные модули

Если кастомные Glue-специфичные модули `invoker` на вход **не принимает** — непокрытая Glue-специфичная часть кита генерируется **Glue-native инструментом**. Manifest фиксирует производителя **каждого** артефакта.

> Проверка `принимает ли invoker кастомные модули` — **verification к этапу реализации**, не опровержение дизайна.

**При трёх плагинах** (§10) build-tool применяется **к kit-слою каждого плагина** независимо: kit каждого плагина собирается своей сборкой, manifest происхождения — per-plugin.

## 10. Декомпозиция доставки — три плагина

**Решение:** форма доставки — не один монолитный плагин, а **три самодостаточных плагина, работающих в любом составе** (7 комбинаций из {П1, П2, П3}). Рантайм-форма (§7–§9) при этом не меняется: любой состав схлопывается к одному `glue` / одной карте / одному контракту.

### 10.1. Три плагина

- **П1 «Правила и знания».** Линзы: Правило, Знание. Цикл: сбор → анализ → **разгрузка** (процедуры выносятся в навыки/skills, узкие правила — в ранбуки со ссылкой в правиле) → визуализация. **Код-судьи (движка вето) нет** — мягкий мир: Правило = проза-директива (концепт §3, требует обдумывания агентом), Знание = справочный корпус. Карта/query/визуализация — есть (общий код, §8.4 / §10.2).
- **П2 «Решения и ограничения».** Линзы: Решение, Ограничение. Цикл: сбор → анализ → **контроль** (мягкий — LLM-судья + новые правила; жёсткий — код-судья + блок) → визуализация. Несёт **движок судьи**, чекеры и сами Решения (источники вето).
- **П3 «Поддержка» (эксплуатация/инфра).** Собственный набор всех линз (Правила/Знания/Навыки/Решения/Ограничения), набитый инфра-контентом, + визуализация. Несёт **движок судьи** и свои инфра-Решения/Ограничения.

**Мягкая проза ≠ жёсткий предикат (концепт §3/§4).** Правило в П1 может предписывать «не делать X», но от этого **не становится** Решением или Ограничением. Жёсткое вето (код-судья) живёт только в П2/П3 и растёт из Решения. Прозаический запрет в правиле — это правило.

**Окружение — ось, не линза (концепт §3).** П3 «Поддержка» — это **инфраструктура-как-тема** (предмет-фасета = infra, тот же набор линз с инфра-контентом), а **не** «плагин окружений». Сама ось `env-scope` (метка local/staging/prod, по которой «на проде блок, на локали ворнинг») едет **с движком судьи в общей дедуплицируемой части** (нужна везде, где есть ограничения — П2, П3), а не отдельным плагином. Не путать ось (метка на проверке) с темой (инфра-контент).

> **Нейминг.** «Поддержка» — ярлык **единицы поставки** (тема = инфра), **не** шестая линза: канон линз остаётся пять (концепт §3). Слаги/идентификаторы плагинов — ASCII (`glossary.md`); нейминг содержания (Правила/Знания/Решения/Ограничения/Поддержка) — отдельно от слагов.

### 10.2. Общие компоненты — встроены и дедуплицируются (newest-**compatible**-wins) — НЕСУЩЕЕ

Общее = **карта** (модель + query layer + визуализация), **общие навыки** (retro, структурирующий), **движок судьи** (там, где есть ограничения — П2, П3; в П1 движка нет). Отдельного «базового пакета» нет — общее **встроено в каждый плагин и дедуплицируется**. Навык, нужный более чем одному плагину, **де-факто общий** (встроен в каждый, дедуп); эксклюзивный — остаётся в плагине.

При установке нескольких плагинов копии общих компонентов схлопываются — но **не** простым newest-wins:

> **Чекпойнт (несущее):** дедуп = **newest-compatible-wins**. Newest-wins допустим **только** при совместимости контрактов. Для дедупа нужен **manifest совместимости** на каждый общий компонент: `{ компонент, версия, contract/API range, потребители }`. Если новейшая копия **несовместима** со старым потребителем (например, новый движок судьи прогнал бы чекер, написанный под несовместимый checker-contract) — это **config error / fail-closed**, а не автоматическая победа newest. Incompatible composition **диагностируется и не активируется** (last-known-good, концепт §5: «несовместимая версия = ошибка конфигурации, а не молчаливый пропуск»).

**Почему не handshake §8.6.** version/hash handshake ловит **build skew** UI↔сервер (равенство хеша). Дедуп **намеренно** создаёт неравенство версий (берёт новейшую) → после дедупа в сессии один движок, handshake тривиально проходит. Он **не** адресует семантическую несовместимость «новый движок × чекер под старый контракт». Эту опасность закрывает manifest совместимости + fail-closed, а не handshake.

### 10.3. Провенанс не пересекает границу самодостаточного плагина — НЕСУЩЕЕ

Провенанс-инвариант концепт §2 (provenance-first): жёсткое вето **всегда** опирается на авторитетный источник. Решения и Ограничения едут вместе в П2 → источник вето всегда под рукой → **П2 самодостаточен даже в одиночку**. То же для П3 (свои инфра-Решения как источники инфра-вето).

**Замечание о расхождении с концепт §2 (фиксируется здесь, концепт не правится).** Концепт §2 допускает авторитетным источником и «**унаследованное правило с зафиксированным происхождением**» — а Правила живут в **П1**. Если бы жёсткое ограничение П2 замыкало load-bearing провенанс на П1-правило, то П2-в-одиночку терял бы источник вето (сирота). Разрешение на уровне формы (вариант **a**):

> **Чекпойнт (несущее):** load-bearing провенанс жёсткого вето **не пересекает границу плагина**, если плагин заявлен самодостаточным. Жёсткие ограничения, поставляемые П2, замыкаются на источник **внутри П2**. Если исходный смысл пришёл из П1-правила — П2 несёт **собственное** Решение/источник, а ссылка на П1-правило остаётся **`references` / lineage** (справочное ребро, концепт §7), **не `derives_from`** (несущее ребро вето). Иначе «П2 самодостаточен в одиночку» не доказан.

### 10.4. Три физических плагина — обоснованы каналом доставки

Три **физических** плагина выбраны **не** потому, что архитектурно невозможен один плагин с тремя выбираемыми бандлами (invoker уже моделирует выбор через `selected_modules` в `modules.json`), а потому что **продуктовая единица установки/докупки — отдельный marketplace-плагин**: маркетплейс Claude Code ставит плагин как целое, «частично установить» бандл из него нельзя. «Докупать по мере боли» (см. Происхождение) = «поставить ещё один плагин» ⇒ нужны отдельные плагины.

Физический разрез — **единственный** источник дедуп-налога (§10.2). Значит:

> **Чекпойнт:** дедуп-налог — **осознанная цена канала дистрибуции**, не свойство архитектуры. Если канал когда-нибудь даст **native selectable bundles** без потери incremental adoption — это может заменить физический разрез **без изменения рантайм-формы** (§7–§9).

## 11. Что остаётся открытым

- **Строгий профиль** (§5) — отложен осознанно; отдельный раунд при необходимости.
- **Содержание контракта CLI** — семейство схем под общей версионированной оболочкой (`versioning.md`); разные операции одной публичной границы: проверки (`событие → вердикт/трасса`) и запросы карты (`запрос → graph result`, тот же query layer §8.4). Конкретные схемы — уровень спеки, не формы. *(Открытый Вопрос 1 раунда контракта — форма оболочки — припаркован: `v4` контракт не меняет.)*
- **Verification к реализации (§9.2):** принимает ли pinned `invoker` кастомные модули — широта покрытия build-tool (per-plugin).
- **Verification к реализации (§10.2):** поддерживает ли механизм плагинов Claude Code (маркетплейс) общие-и-дедуплицируемые компоненты между отдельно установленными плагинами, или дедуп реализуется **newest-compatible-wins резолвером поверх физически встроенных копий**. Дизайн корректен при любом ответе (паттерн §9.2); меняется лишь способ реализации дедупа.
- **Навигационная ссылка концепт §12** сейчас указывает на `v3`. По процесс-инварианту концепт-спец в этом раунде **не правится**; переключение указателя §12 → `v4` — **отдельное решение оператора** (как делалось при v2→v3), вне скоупа этой формы.

Базовая **форма** продукта определена: рантайм-гибрид (кит + `glue` CLI + адаптеры + визуальный клиент) с одной публичной интеграционной границей (контракт CLI) и визуализацией карты в поставке; **доставка — три самодостаточных плагина (П1/П2/П3) в любом составе**, общее встроено-и-дедуплицировано по newest-compatible-wins, провенанс вето не пересекает границу самодостаточного плагина, физический разрез — осознанная цена marketplace-канала.
