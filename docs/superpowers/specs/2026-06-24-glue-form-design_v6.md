# Glue — форма продукта (design)

**Дата:** 2026-06-24
**Статус:** рабочий design формы — упаковка, клиенты, человеческое представление карты, декомпозиция доставки. Ведётся поверх застывшей концепции (`2026-06-23-glue-concept-design_v2.md`); концепцию не переписывает.
**Модель:** Glue помогает избегать честных ошибок кооперативного агента, не защищает от намеренного обхода.

---

## Корневая формула — помогаем избегать ошибок

**Целевой актор сотрудничает.** Glue помогает кооперативному агенту (и человеку) **не ошибаться**: не забыть правило, не потерять контекст, не пойти не туда. **Намеренный обход — вне модели**: это твой проект, ты его хозяин; строить оборону против самого себя бессмысленно. Локальные проверки — **guardrails (ремень безопасности + пищалка), не граница безопасности**. Это единственное место, где проговаривается «помощь, не оборона»; ниже оно подразумевается.

**Что Glue делает:** ловит честные ошибки в момент действия + **честно диагностирует непроверенное** («тут я не проверил»).

**Гибрид упаковки выводится из этой модели** (не выбирается из меню):

- **локальный лёгкий инструмент** — проверки в момент работы (агент дёргает через хук); ловит ошибки;
- **платформенный контур** (GitHub / CI / branch protection) — для **наружных действий** (push / merge / deploy): сильнее защищает от **случайного** обхода и существует уже сейчас.

**Источник правды — файлы.** Состояние инструмента производно и пересобираемо; второй авторитетной БД нет (компиляция файлов → валидированный снимок + индекс + last-known-good).

**Сердцевина проверок** — машинная, ради **надёжной помощи**: валидация набора и атомарная активация; разрешение назначений; фиксация снимка доказательств; запуск чекеров; агрегация + fail-closed для блокирующих; машинная конфликт-детекция; трасса и last-known-good.

---

## Что должно быть исправно, чтобы помощник не врал

Перечень того, что должно **работать корректно**, иначе сигнал помощника станет ложным.

- **Предсказуемость (детерминизм):** одинаковая ситуация → одинаковый ответ. Помощник, который «то ловит, то нет», бесполезен. Вход = канонический снимок; время/сеть/случайность карантинятся в фазе сбора (концепт, раздел «Судьи и проверки»).
- **Честность непокрытого:** если проверка не сработала или класс действия не покрыт — сказать прямо, **не** создавать иллюзию контроля. Худшее для помощника — ложное спокойствие.
- **Сборщик собирает честно ↔ реальность недоверена:** снимок доказательств обеспечивает воспроизводимость (помощник показывает ровно то, что видел сборщик).
- **Останов при поломке проверки на опасном (fail-closed для блокирующих):** если чекер упал — лучше притормозить и сказать «не смог проверить», чем молча пропустить. last-known-good: невалидный набор не применяется, действует предыдущий валидный.

## Карта проверок — что когда проверяется

Действие набирает проверки от своих признаков; **все применимые проверки должны пройти** (конъюнкция — не теряем ни одной). Два рода машинной проверки:

- **блокирующая** — fail-closed, останавливает действие;
- **сигнальная** — мягкая, деградирует, подсказывает, **не** останавливает.

| Признак действия | Что добавляется |
|---|---|
| ø (обычная нечувствительная правка) | ничего; свободная работа |
| чтение / навигация / карта / анализ | ничего (by design) |
| под ограничением | блокирующая проверка (рантайм-судья) |
| наружу (публичный PR, публикация, деплой) | **operator-gate** — отдельный гейт, буквальное «да» |
| разрушительное / высокорисковое (force-push, reset --hard, удаление) | **operator-gate** — отдельный гейт каждый раз, форма **report+STOP** |
| меняет несущее звено модели | блокирующая проверка + подтверждение по типу перехода |
| промоция → Правило | подтверждение нормы; структурная проверка — сигнальная |
| промоция → Ограничение | разворачивается в «меняет несущую модель» |

**`operator-gate` и audit-trail — дисциплина осознанного действия:** помогают человеку *не сделать наугад* то, что заслуживает паузы.

### Изменение несущей модели — осознанные фазы

**Критерий несущего звена** (не закрытый перечень): несущее — **любое управляемое звено, изменение которого меняет вычисление, доставку или применение вердикта**. Примеры: само ограничение, чекер, источник (provenance), назначение. Согласовано с осью «несущести» (концепт, раздел «Типология рёбер»).

```
авторизация перехода → правка → проверка набора
  → нужные подтверждения → атомарная активация
```

Подтверждения задаёт тип перехода: новое вето — G1 (норма) → G2 (включение); ослабление/снятие — teardown-гейт; техническая замена без смены смысла — простая авторизация правки.

## Где что проверяется

- **Локально, в момент работы — guardrails.** Лёгкий инструмент на машине, агент вызывает через хук. Ловит ошибки агента.
- **На платформе — для наружных действий.** GitHub / CI / branch protection сильнее защищают от **случайного** обхода push / merge / deploy. Эта защита уже существует — Glue на неё опирается, а не дублирует.

Сила гарантии — **«ловит ли типичную ошибку»**; для повседневных локальных действий лёгкого guardrail достаточно.

---

## Требования к упаковке

- **лёгкий локальный инструмент**, вызываемый агентом через хук, для проверок в момент работы;
- **опора на платформенный контур** (GitHub / CI) для наружных действий — не дублировать;
- **файлы — источник правды**, состояние инструмента производно и пересобираемо;
- **детерминизм** проверок (предсказуемый сигнал);
- **честная диагностика непокрытого**;
- **fail-closed для блокирующих** на опасном.

## Рантайм-форма — контракт CLI

**Внешняя граница — процессный контракт CLI.** Все *публичные* интеграции вызывают Glue как **команду**; никто не обязан импортировать его код. (Уточнение границы при появлении браузерного клиента — ниже, «Человеческое представление карты».)

**Логически рантайм Glue — три вещи (+ визуальный клиент):**

1. **набор артефактов** — правила / навыки / шаблоны / знания (для Claude Code, Codex, Gemini);
2. **`glue`** — исполняемая команда: судьи + запросы к карте. Внутри может быть кодовая библиотека ради тестируемости, но это **внутренняя архитектура, не публичный способ интеграции**;
3. **адаптеры** — тонкая привязка событий конкретного агента или CI к команде («тупой адаптер» — без нормативной логики).

Эта тройка — **логический** состав рантайма. Как она пакуется в плагины — см. «Декомпозиция доставки».

Продукт гибридный по составу, но имеет **одну публичную интеграционную границу — контракт CLI**.

**Контракт (структурированный, версированный):**

- событие → **JSON в stdin**;
- вердикт + трасса → **JSON в stdout** (`--format=json`);
- человекочитаемое представление того же ответа → **stdout** (`--format=human`);
- диагностический текст → **stderr** (только диагностика, не ответ);
- итог дополнительно закреплён **кодом выхода**;
- интерактивный текстовый вывод — **представление того же ответа**, не отдельный путь.

Claude Code, Codex, Gemini и CI используют **один** контракт `событие → glue → структурированный вердикт`; различаются **только адаптеры**.

> **Доставка и онбординг — отдельный слой.** Плагины **разворачивают** ядро (набор артефактов + CLI + адаптеры + визуальный клиент), но **не являются** им и **не запирают** его. Команда `glue` и набор артефактов самостоятельны.

## Человеческое представление карты

Концепт (раздел «Карта») задал модель: одна каноническая графовая модель → два потребителя (агент = запросный движок; человек = навигационное представление, read-only к модели, «read-only ≠ статичная картинка»).

### Визуализация — в базовой поставке

Визуальное представление карты — **важное представление в базовой поставке**, а не вспомогательный довесок. Но и **не суть продукта**: суть — связанная адресуемая структура артефактов (это и есть главный guardrail, концепт «Что такое Glue»); карта её **показывает**. Текстовый суррогат (ASCII-дерево / таблицы CLI) для навигации по графу малоинформативен и **отвергнут** как «приличная карта».

Требование — **интерактивный клиент, не статичное изображение** (раскрытие соседей, выбор трассы, фильтры по видам, переход к источнику). Статичность *доставки* клиента (web-ресурсы могут быть статическими файлами) этому не противоречит — речь о статичности *представления*. Физически клиент — в core-плагине (см. «Декомпозиция доставки»).

### Упаковка — `B`, направление зависимостей — `A`

- **по упаковке — `B`:** `glue map --serve` запускает локальное полотно; **один установленный продукт, одна команда**;
- **по направлению зависимостей — `A`:** `core/query` и `web-client` — **раздельные модули**; ядро **не импортирует** UI; CLI — **composition root**.

### Архитектурная схема

```
browser UI → session API → query layer → canonical model
```

Сервер **локальный, сессионный, read-only** — мутационных методов у него нет (мутация вынесена за границу карты).

### Query layer — единственный traversal

Вся **семантическая навигация** — в query layer: provenance, impact, соседство, виды. И CLI-`--format`, и session API — **два транспорта над одним query layer**; UI **не реализует** параллельную графовую логику (только layout, pan/zoom, подсветка).

> **Инвариант полноты:** query layer обязан покрывать **все** навигационные операции человека. Web-клиент **не должен требовать изменений** канонической модели или query layer.

> Query layer — общий рантайм (в core-плагине). При составе `core + один П1` он **чисто деградирует** к графу без узлов-ограничений (нет вето → нет вето-узлов).

### Контракт потоков

- `--format=json` → структурированный ответ в **stdout**;
- `--format=human` → человекочитаемая карта в **stdout**;
- **stderr** → только диагностика.

### Граница

> **CLI — единственная *публичная интеграционная* точка.** `map --serve` открывает **внутренний сессионный транспорт** только для комплектного UI: непубличный, без внешних гарантий совместимости; внутренний version/hash handshake ловит рассинхрон сборки; эфемерный порт + токен сессии изолируют *сессию*. UI и сервер живут в одном core-плагине → рассинхрон маловероятен, handshake — дешёвая страховка.

## Декомпозиция доставки — core-плагин + контент-паки

**Решение:** общий рантайм вынесен в отдельный **core-плагин `glue-core`** (в маркетплейсе `glue`), а П1/П2/П3 — **контент-паки**, объявляющие зависимость от него (`dependencies`). Платформа Claude Code авто-доставляет core при установке любого пака. Пользователь по-прежнему ставит «нужный плагин по мере боли»; core приезжает невидимо.

**Почему так, а не «общее встроено в каждый пак + самодедуп».** Документация Claude Code: плагины **полностью изолированы** (каждый в своей кэш-папке `~/.claude/plugins/cache/...`; ссылки на файлы вне папки плагина запрещены) и общее между отдельно установленными плагинами **не дедуплицируется**. Самодельный рантайм-резолвер копий боролся бы с платформой. Нативные **зависимости + version-constraints** дают то же штатно.

Контент-паки следуют **порядку прихода боли**: правила тонут → **П1**; решения надо контролировать → **П2**; появляется эксплуатация → **П3**. Каждая волна накрывает меньше проектов; **П1 и П2 — первыми, П3 — в беклоге**.

### Состав

- **`glue-core`** (core) — рантайм в единственном экземпляре: команда `glue`, **карта / query layer / визуализация**, **судьи** (кодовый — детерминированный, вето; **LLM-судья** — рекомендательный, без вето, ловит смысловые отклонения рано и широко), **адаптеры** (тонкие хуки, переводящие события агентов/CI в вызов `glue`), общие навыки (retro, структурирующий).
- **`glue-rules`** (П1 «Правила и знания») — контент: виды Правило, Знание. Цикл: сбор → анализ → разгрузка (процедуры → навыки, узкие правила → ранбуки) → визуализация. Своих ограничений не приносит → код-судье нечего ветировать (мягкий мир: работают LLM-судья и видимость).
- **`glue-decisions`** (П2 «Решения и ограничения») — контент: виды Решение, Ограничение + чекеры + сами Решения (источники вето).
- **`glue-support`** (П3 «Поддержка») — контент: весь набор пяти видов с инфра-контентом + свои инфра-Решения/Ограничения. **В беклоге** — после П1+П2.

Каждый контент-пак объявляет `dependencies: [{ name: "glue-core", version: "~0.1.0" }]`. Состав поставки — **`glue-core` + любой набор {`glue-rules`, `glue-decisions`, `glue-support`}**.

**Судьи — в core** (не в паках): одна копия. «П1 без код-судьи» = П1 не приносит ограничений → код-судье при `core + П1` нечего ветировать; мягкий контроль (LLM-судья + видимость структуры) при этом работает.

> **Хук и судьи — оба в core, между ними границы плагина нет:** хук core зовёт `glue` из той же core-папки. Изоляция плагинов тут ничего не ломает. Граница всплывает в другом — см. «Что остаётся открытым».

**Мягкая проза ≠ жёсткий предикат (концепт «Судьи и проверки»).** Правило в П1 может предписывать «не делать X», но Решением/Ограничением не становится; жёсткое вето растёт только из Решения (П2/П3).

**Окружение — ось, не вид (концепт «Артефакты»).** П3 — **инфраструктура-как-тема** (тема = infra, те же пять видов с инфра-контентом), не «плагин окружений». Ось `env-scope` — часть судьи (в core). Не путать ось (метка на проверке) с темой (инфра-контент).

> **Нейминг.** «Поддержка» — ярлык контент-пака (тема = инфра), **не** шестой вид: канон видов — пять. Слаги плагинов — ASCII (`glossary.md`).

### Провенанс не пересекает границу контент-пака — НЕСУЩЕЕ

Это **применение инварианта концепта** («источник доступен там, где ограничение действует», раздел «Происхождение прежде всего») к границам плагинов. core даёт **механизм** (судьи), но **не авторитет** — источники вето (Решения) едут в контент-паке. Решения и Ограничения вместе в П2 → источник вето всегда под рукой → **`core + П2` самодостаточны по провенансу** (источник вето лежит внутри пака, не в чужом). То же для П3. Это про провенанс, **не** про работоспособность: чтобы вето реально сработало, core должен суметь прочитать контент пака — а это пока **открытый несущий риск** (см. «Что остаётся открытым»).

> **Чекпойнт (несущее):** load-bearing провенанс жёсткого вето **не пересекает границу контент-пака**. Жёсткие ограничения П2 замыкаются на источник **внутри П2**. Если исходный смысл пришёл из П1-правила — П2 несёт **собственное** Решение/источник, а ссылка на П1-правило остаётся **`references` / `promoted_from`** (родословная), **не `derives_from`**. Иначе ограничение, чей источник в другом (возможно не установленном) паке, теряет легитимность.

### Совместимость версий — нативная

Контент-пак объявляет требуемую совместимую линию core через version-constraint (`dependencies: [{ name: "glue-core", version: "~0.1.0" }]` — tilde фиксирует минорную линию: `>=0.1.0 <0.2.0`). Платформа разрешает установку и порядок; несовместимость `core ↔ пак` ловится **на установке**, а не молчаливым прогоном чекера под чужой контракт. Самодельный manifest совместимости и newest-compatible-wins резолвер **не нужны** — их роль выполняют нативные зависимости. Принцип концепта («несовместимая версия = ошибка конфигурации, а не молчаливый пропуск») сохраняется, обеспечивается платформой.

> **Платформенные предпосылки (подтверждено [docs](https://code.claude.com/docs/en/plugin-dependencies)).** Одного поля `dependencies` достаточно не безусловно — есть пороги версий Claude Code:
> - version-constraints в `dependencies` — **с v2.1.110**;
> - согласованный enable/disable зависимостей (включил пак → включился `glue-core`; нельзя выключить `glue-core`, пока включён зависящий пак) — **с v2.1.143** (раньше — только именованный плагин + ошибка `dependency-unsatisfied` на следующей загрузке);
> - `claude plugin prune` (убрать осиротевшие авто-зависимости) — **с v2.1.121**.
>
> Зависимость по умолчанию резолвится **в том же маркетплейсе**, что и объявляющий плагин. Все четыре плагина живут в маркетплейсе `glue`, поэтому cross-marketplace allowlist (`allowCrossMarketplaceDependenciesOn`) **не нужен**. Версии резолвятся по git-тегам формата **`{plugin-name}--v{version}`** (напр. `glue-core--v0.1.0`; тег ставится `claude plugin tag --push`). Без тега зависимый пак отключается с `no-matching-tag` — значит **тегирование релизов `glue-core` обязательно**, иначе автодоставка по версии не работает.

### Физический разрез — подтверждён каналом доставки

Документация Claude Code: единица установки = **плагин целиком**; выбираемых под-бандлов внутри одного плагина нет. Значит incremental adoption = **отдельные плагины** — подтверждённый факт. core-плагин + контент-паки реализует «докупать по мере боли» через нативные зависимости; **дублирования между паками не возникает** — общее в core в единственном экземпляре.

---

## Что остаётся открытым

- **Содержание контракта CLI** — семейство схем под общей версионированной оболочкой (`versioning.md`): проверки (`событие → вердикт/трасса`) и запросы карты (`запрос → graph result`, тот же query layer). Внутри — **открытый Вопрос 1**: одна дискриминированная оболочка (A) или раздельные контракты по операции (B). Уровень спеки, не формы.
- **Несущая verification — как core читает контент паков.** Судья в core; ограничения/чекеры — в контент-паках (их кэш-папки). Запрет «плагин не ссылается на файлы вне своей папки» — про **упаковку**, не про рантайм: запущенный `glue` читает кэш других плагинов и подхватывает контент. **Но если платформа песочит исполнение так, что core-процесс не достучится до папок паков — модель «судьи в core + контент в паках» придётся менять.** Это главный риск к проверке на реальной платформе. Плюс точный механизм, которым хуки зависимых паков зовут `glue` из core (PATH / MCP / command), и сосуществование версий core.

> **Закрыто проверкой документации:** единица установки = плагин целиком (нет под-бандлов) → отдельные плагины оправданы; платформа не дедуплицирует и изолирует плагины → дедуп снят выносом общего в core.

Базовая **форма** определена **с одной оговоркой**: рантайм-гибрид (набор артефактов + `glue` CLI + адаптеры + визуальный клиент) с одной публичной интеграционной границей (контракт CLI) и визуализацией карты в поставке; **доставка через маркетплейс `glue` — core-плагин `glue-core` (команда + карта + судьи + адаптеры + общие навыки) и контент-паки `glue-rules` / `glue-decisions` / `glue-support` с зависимостью от `glue-core`; П1+П2 первыми, П3 в беклоге**; общее живёт в core в единственном экземпляре (платформенные зависимости вместо самодедупа), провенанс вето не пересекает границу контент-пака. **Оговорка:** несущий механизм «core читает контент паков» ещё не подтверждён прототипом — пока он открыт, форму нельзя считать закрытой к реализации (см. «Что остаётся открытым»).
