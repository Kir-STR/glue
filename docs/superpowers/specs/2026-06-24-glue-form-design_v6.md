**Дата:** 2026-06-24

Основание — проверка документации Claude Code (см. §9, §10): плагины полностью изолированы и не дедуплицируются платформой. `v5` уходит в архив. Ведётся поверх застывшей концепции (`2026-06-23-glue-concept-design_v2.md`); концепцию не переписывает.
**Скоуп:** форма продукта Glue — упаковка, клиенты, человеческое представление карты, декомпозиция доставки.
**Происхождение:** раунд формы. `v1` — модель «защита от обхода»; `v2` сузил до «помогаем избегать честных ошибок»; `v3` закрыл упаковку (контракт CLI + визуализация карты); `v4` зафиксировал три плагина; `v5` — чистка под концепт v2; `v6` — доставка через core-плагин + зависимости (после проверки доков маркетплейса).

Контент-паки следуют **порядку, в котором приходит боль** у растущего проекта: (1) правила тонут, нужна прополка и видимость → **П1**; (2) решения нужно фиксировать и контролировать исполнение → **П2**; (3) появляется эксплуатация со своей спецификой → **П3**. Каждая следующая волна накрывает меньше проектов; **П1 и П2 — первыми, П3 — в беклоге**. Все паки сидят на общем core-плагине, который приезжает зависимостью.

**Процесс-инвариант:** форма не переписывает концепцию. Расхождения с концепт-спецем фиксируются замечанием здесь (см. §9.2); концепт-спец и `IDEA.md` не правятся.

**Метод:** проза, по одному решению, взаимная адъюдикация.

## 1. Корневая формула — «помогаем избегать ошибок»

**Целевой актор сотрудничает.** Glue помогает кооперативному агенту (и человеку) **не ошибаться**: не забыть правило, не потерять контекст, не пойти не туда. **Намеренный обход — вне модели**: это твой проект, ты его хозяин и можешь сделать с ним что угодно; строить оборону против самого себя бессмысленно.

Отсюда — что Glue делает и чего не делает:

- **делает:** ловит честные ошибки в момент действия + **честно диагностирует непроверенное** («тут я не проверил»);
- **не делает:** не препятствует тому, кто *хочет* обойти; локальные проверки — **guardrails (ремень безопасности + пищалка), не граница безопасности**.

**Гибрид упаковки выводится из этой модели** (не выбирается из меню):

- **локальный лёгкий инструмент** — проверки в момент работы (агент дёргает через хук); ловит ошибки, обходимость не важна;
- **платформенный контур** (GitHub / CI / branch protection) — для **наружных действий** (push / merge / deploy): даёт более сильную защиту от **случайного** обхода и существует уже сейчас.

**Источник правды — файлы.** Состояние инструмента производно и пересобираемо; второй авторитетной БД нет (компиляция файлов → валидированный снимок + индекс + last-known-good).

**Сердцевина проверок** сохраняется как машинная — но ради **надёжной помощи**, не неуязвимости: валидация набора и атомарная активация; разрешение назначений; фиксация снимка evidence; запуск чекеров; агрегация + fail-closed для блокирующих; машинная конфликт-детекция; трасса и last-known-good.

## 2. Что должно быть исправно, чтобы помощник не врал

Перечень того, что должно **работать корректно**, иначе сигнал помощника станет ложным.

- **Предсказуемость (детерминизм):** одинаковая ситуация → одинаковый ответ. Помощник, который «то ловит, то нет», бесполезен. Вход = канонический снимок; время/сеть/случайность карантинятся в фазе сбора (концепт-спец §5).
- **Честность непокрытого:** если проверка не сработала или класс действия не покрыт — сказать прямо, **не** создавать иллюзию контроля. Худшее для помощника — ложное спокойствие.
- **Сборщик собирает честно ↔ реальность недоверена:** снимок evidence обеспечивает воспроизводимость (помощник показывает ровно то, что видел сборщик). Это нужно для предсказуемости и честности.
- **Останов при поломке проверки на опасном (fail-closed для блокирующих):** если чекер упал — лучше притормозить и сказать «не смог проверить», чем молча пропустить и внушить, что всё ок. last-known-good: невалидный набор не применяется, продолжает действовать предыдущий валидный.

## 3. Карта проверок — что когда проверяется

Действие набирает проверки от своих признаков; **все применимые проверки должны пройти** (конъюнкция — не теряем ни одной, иначе пропустим ошибку). Два рода машинной проверки:

- **блокирующая (enforcing-M)** — fail-closed, останавливает действие;
- **сигнальная (advisory-M)** — мягкая, деградирует, подсказывает, **не** останавливает.

| Признак действия | Что добавляется |
|---|---|
| ø (обычная нечувствительная правка) | ничего; свободная работа |
| чтение / навигация / карта / анализ | ничего (by design) |
| под ограничением | блокирующая проверка (рантайм-судья) |
| наружу (outward-facing: публичный PR, публикация, деплой) | **operator-gate** — отдельный гейт, буквальное «да» |
| разрушительное / высокорисковое (force-push, reset --hard, удаление) | **operator-gate** — отдельный гейт каждый раз, форма **report+STOP** |
| меняет несущее звено модели | блокирующая проверка + осознанное подтверждение по типу перехода (§3.1) |
| промоция → Правило | подтверждение нормы; структурная проверка — сигнальная |
| промоция → Ограничение | разворачивается в «меняет несущую модель» |

**`operator-gate` и audit-trail — дисциплина осознанного действия, не защита от злоумышленника.** Они помогают человеку *не сделать наугад* то, что заслуживает паузы.

### 3.1. Изменение несущей модели — осознанные фазы

**Критерий несущего звена** (не закрытый перечень): несущее — **любое управляемое звено, изменение которого меняет вычисление, доставку или применение вердикта**. Примеры: само ограничение, чекер, источник (provenance), назначение. Критерий согласован с осью «несущести» концепт-спец §7.

```
авторизация перехода → правка → проверка набора
  → нужные подтверждения → атомарная активация
```

Подтверждения задаёт тип перехода (осознанность, не оборона): новое вето — G1 (норма) → G2 (включение); ослабление/снятие — teardown-гейт; техническая замена без смены смысла — простая авторизация правки.

## 4. Где что проверяется

- **Локально, в момент работы — guardrails.** Лёгкий инструмент на машине, агент вызывает через хук. Ловит ошибки агента. Обходимость не важна и не цель.
- **На платформе — для наружных действий.** GitHub / CI / branch protection дают более сильную защиту от **случайного** обхода push / merge / deploy. Эта защита уже существует — Glue на неё опирается, а не дублирует.

Сила гарантии — **«ловит ли типичную ошибку»**; для повседневных локальных действий лёгкого guardrail достаточно.

---

## 5. Требования к упаковке

- **лёгкий локальный инструмент**, вызываемый агентом через хук, для проверок в момент работы;
- **опора на платформенный контур** (GitHub / CI) для наружных действий — не дублировать;
- **файлы — источник правды**, состояние инструмента производно и пересобираемо;
- **детерминизм** проверок (предсказуемый сигнал);
- **честная диагностика непокрытого** (не врать, что проверено);
- **fail-closed для блокирующих** на опасном (останов вместо молчаливого пропуска).

## 6. Рантайм-форма — контракт CLI (перенос из v3)

**Внешняя граница — процессный контракт CLI (вариант A).** Все *публичные* интеграции вызывают Glue как **команду**; никто не обязан импортировать его код. (Уточнение границы при появлении браузерного клиента — §7.6.)

**Логически рантайм Glue — три вещи (+ визуальный клиент, §7):**

1. **кит артефактов** — набор правил / навыков / шаблонов / знаний (для Claude Code, Codex, Gemini);
2. **`glue`** — исполняемая команда: кодовые судьи + запросы к карте. Внутри может быть кодовая библиотека ради тестируемости, но это **внутренняя архитектура, не публичный способ интеграции**;
3. **адаптеры** — тонкая привязка событий конкретного агента или CI к команде («тупой адаптер» — без нормативной логики).

Эта тройка — **логический** состав рантайма. Как она пакуется в плагины (что в core, что в контент-паках) — §9.

Продукт гибридный по составу, но имеет **одну публичную интеграционную границу — контракт CLI**.

**Контракт (структурированный, версированный):**

- событие → **JSON в stdin**;
- вердикт + трасса → **JSON в stdout** (`--format=json`);
- человекочитаемое представление того же ответа → **stdout** (`--format=human`);
- диагностический текст → **отдельно (stderr)**; stderr — **только** диагностика, не ответ;
- итог дополнительно закреплён **кодом выхода**;
- интерактивный текстовый вывод — **представление того же ответа**, не отдельный путь.

Claude Code, Codex, Gemini и CI используют **один** контракт `событие → glue → структурированный вердикт`; различаются **только адаптеры**.

> **Доставка и онбординг — отдельный слой (§8, §9).** Плагины **разворачивают** ядро (кит + CLI + адаптеры + визуальный клиент), но **не являются** им и **не запирают** его. Команда `glue` и кит самостоятельны.

## 7. Человеческое представление карты — визуализация в базовой поставке (перенос из v3)

Концепт-спец §6 задал модель: одна каноническая графовая модель → два потребителя (агент = запросный движок; человек = навигационное представление, read-only к модели, «read-only ≠ статичная картинка»).

### 7.1. Визуализация входит в базовую поставку

Визуальное представление карты — **ключевой смысл продукта**, не вспомогательный слой. Текстовый суррогат (ASCII-дерево / таблицы CLI) малоинформативен для навигации по графу и **отвергнут** как «приличная карта». Интерактивное визуальное полотно входит в **базовую** поставку (физически — в core-плагине, §9).

Требование — **интерактивный клиент, не статичное изображение** (раскрытие соседей, выбор трассы, фильтры по видам, переход к источнику). Статичность *доставки* клиента (web-ресурсы могут быть статическими файлами) этому не противоречит — речь о статичности *представления*.

### 7.2. Упаковка — `B`, направление зависимостей — `A`

- **по упаковке — `B`:** `glue map --serve` запускает локальное полотно; **один установленный продукт, одна команда**;
- **по направлению зависимостей — `A`:** `core/query` и `web-client` — **раздельные модули**; ядро **не импортирует** UI; CLI выступает **composition root**.

### 7.3. Архитектурная схема

```
browser UI → session API → query layer → canonical model
```

Сервер **локальный, сессионный, read-only** — мутационных методов у него нет (мутация вынесена за границу карты, концепт §6: gated workflow).

### 7.4. Query layer — единственный traversal; разделение ролей

Вся **семантическая навигация** выполняется в query layer: provenance, impact, соседство, виды. И CLI-`--format`, и session API — **два транспорта над одним query layer**; UI **не реализует** параллельную графовую логику. UI вправе выполнять **только представление**: layout, pan/zoom, локальную подсветку.

> **Инвариант полноты** (висит на query layer, не на формате): query layer обязан покрывать **все** навигационные операции человека. Web-клиент **не должен требовать изменений** канонической модели или query layer.

> **Карта/query/визуализация — в core-плагине (§9.1).** Это **общий рантайм**, единая копия; контент-паки его не несут. При составе `core + один П1` query layer **чисто деградирует** к графу без узлов-ограничений (нет вето → нет вето-узлов), а не повисает на отсутствующих узлах.

### 7.5. Контракт потоков

- `--format=json` → структурированный ответ в **stdout**;
- `--format=human` → человекочитаемая карта в **stdout**;
- **stderr** → только диагностика.

### 7.6. Граница

> **CLI — единственная *публичная интеграционная* точка.** `map --serve` открывает **внутренний сессионный транспорт** только для комплектного UI.

Свойства session API:

- **непубличный, без внешних гарантий совместимости** (CLI-контракт версионируется, `versioning.md`);
- **внутренний version/hash handshake:** комплектный UI и сервер обязаны **обнаруживать рассинхрон сборки** — устаревшая вкладка против свежего сервера падает явно, а не тихо неверно рендерит (иначе нарушается «честность непокрытого» §2);
- **эфемерный порт + токен сессии** изолируют *сессию*, а не объявляют публичный контракт.

UI и сервер живут в одном core-плагине → рассинхрон сборки между ними маловероятен, но handshake остаётся дешёвой страховкой.

## 8. Сборка кита (деталь реализации)

Установщик Glue раскладывает кит **сам**; в момент установки/работы внешний build-tool не нужен. На этапе **сборки релиза** Glue может **переиспользовать код `invoker`** (соседний file-layer scaffolder) для той части кита, что он и так умеет раскладывать; непокрытое генерируется **своим** кодом. Какой код собрал какой кусок — фиксирует manifest происхождения. Это **деталь реализации, не вопрос формы**.

## 9. Декомпозиция доставки — core-плагин + контент-паки

**Решение:** общий рантайм вынесен в отдельный **core-плагин `glue`**, а П1/П2/П3 — **контент-паки**, объявляющие зависимость от него (`dependencies`). Платформа Claude Code авто-доставляет core при установке любого пака. Пользователь по-прежнему ставит «нужный плагин по мере боли»; core приезжает невидимо.

**Почему так, а не «общее встроено в каждый пак + самодедуп».** Документация Claude Code: плагины **полностью изолированы** (каждый в своей кэш-папке `~/.claude/plugins/cache/...`; ссылки на файлы вне папки плагина запрещены) и общее между отдельно установленными плагинами **не дедуплицируется**. Самодельный рантайм-резолвер копий боролся бы с платформой. Нативные **зависимости + version-constraints** дают то же самое штатно (verification §10 — подтверждено доками).

### 9.1. Состав

- **`glue` core** — рантайм в единственном экземпляре: команда `glue` (кодовые судьи + запросы к карте), **карта / query layer / визуализация**, **движок судьи**, **адаптеры** (тонкие хуки, переводящие события агентов/CI в вызов `glue`), общие навыки (retro, структурирующий).
- **П1 «Правила и знания»** — контент: виды Правило, Знание. Цикл: сбор → анализ → разгрузка (процедуры → навыки, узкие правила → ранбуки со ссылкой) → визуализация. Своих ограничений не приносит → общему судье нечего ветировать (мягкий мир). `dependencies: [glue-core]`.
- **П2 «Решения и ограничения»** — контент: виды Решение, Ограничение + чекеры + сами Решения (источники вето). `dependencies: [glue-core]`.
- **П3 «Поддержка»** — контент: весь набор пяти видов с инфра-контентом + свои инфра-Решения/Ограничения. **В беклоге** — после П1+П2. `dependencies: [glue-core]`.

Состав поставки — **core + любой набор {П1, П2, П3}**. Порядок зрелости: core + П1 + П2 первыми, П3 позже.

**Движок судьи — в core** (не в паках): одна копия, никакого дедупа между П2 и П3. «П1 без код-судьи» = П1 не приносит ограничений → общему движку при `core + П1`-составе нечего ветировать (дремлет).

**Мягкая проза ≠ жёсткий предикат (концепт §3/§4).** Правило в П1 может предписывать «не делать X», но Решением/Ограничением не становится; жёсткое вето (код-судья) растёт только из Решения (П2/П3).

**Окружение — ось, не вид (концепт §3).** П3 — **инфраструктура-как-тема** (предмет-фасета = infra, те же пять видов с инфра-контентом), не «плагин окружений». Ось `env-scope` — часть движка судьи (в core), нужна везде, где есть ограничения. Не путать ось (метка на проверке) с темой (инфра-контент).

> **Нейминг.** «Поддержка» — ярлык контент-пака (тема = инфра), **не** шестой вид: канон видов — пять (концепт §3). Слаги/идентификаторы плагинов — ASCII (`glossary.md`).

### 9.2. Провенанс не пересекает границу контент-пака — НЕСУЩЕЕ

Провенанс-инвариант концепт §2 (provenance-first): жёсткое вето **всегда** опирается на авторитетный источник. core даёт **механизм** (движок судьи), но **не авторитет** — источники вето (Решения) едут в контент-паке. Решения и Ограничения вместе в П2 → источник вето всегда под рукой → **`core + П2` самодостаточны даже в одиночку**. То же для П3.

**Замечание о расхождении с концепт §2 (фиксируется здесь, концепт не правится).** Концепт §2 допускает авторитетным источником и «**унаследованное правило с зафиксированным происхождением**» — а Правила живут в **П1**. Если бы жёсткое ограничение П2 замыкало load-bearing провенанс на П1-правило, то `core + П2` без П1 терял бы источник вето (сирота). Разрешение (вариант **a**):

> **Чекпойнт (несущее):** load-bearing провенанс жёсткого вето **не пересекает границу контент-пака**. Жёсткие ограничения, поставляемые П2, замыкаются на источник **внутри П2**. Если исходный смысл пришёл из П1-правила — П2 несёт **собственное** Решение/источник, а ссылка на П1-правило остаётся **`references` / lineage** (концепт §7), **не `derives_from`** (несущее ребро вето).

### 9.3. Совместимость версий — нативная

Контент-пак объявляет требуемый мажор core через version-constraint (`dependencies: [{ name: "glue-core", version: "~X.Y" }]`). Платформа разрешает установку и порядок; несовместимость `core ↔ пак` ловится **на установке**, а не молчаливым прогоном чекера под чужой контракт.

> **Чекпойнт:** самодельный manifest совместимости и newest-compatible-wins резолвер (были в `v4`/`v5`) **не нужны** — их роль выполняют нативные зависимости + version-constraints. Принцип концепт §5 («несовместимая версия = ошибка конфигурации, а не молчаливый пропуск») сохраняется, но обеспечивается платформой.

### 9.4. Физический разрез — подтверждён каналом доставки

Документация Claude Code: единица установки = **плагин целиком**; выбираемых под-бандлов внутри одного плагина нет. Значит incremental adoption = **отдельные плагины** — это **подтверждённый факт**, а не предпосылка (`v4`/`v5` помечали как непроверенное). core-плагин + контент-паки реализует «докупать по мере боли» через нативные зависимости; **дедуп-налога больше нет** — общее в core в единственном экземпляре, дублирования между паками не возникает.

## 10. Что остаётся открытым

- **Содержание контракта CLI** — семейство схем под общей версионированной оболочкой (`versioning.md`): проверки (`событие → вердикт/трасса`) и запросы карты (`запрос → graph result`, тот же query layer §7.4). Внутри — **открытый Вопрос 1**: одна дискриминированная оболочка (A) или раздельные контракты по операции (B). Уровень спеки, не формы.
- **Verification к реализации (§9):** точный механизм, которым **core-плагин отдаёт рантайм `glue`** хукам зависимых контент-паков (PATH / MCP server / command) — доки подтверждают зависимости и изоляцию, но детально это не описывают. Плюс сосуществование версий core, если контент-паки потребуют разные мажоры (разрешается version-constraints). Дизайн корректен при штатном поведении механизма зависимостей.

> **Закрыто проверкой документации (`v6`):** единица установки = плагин целиком (нет под-бандлов) → отдельные плагины оправданы; платформа не дедуплицирует и изолирует плагины → дедуп снят выносом общего в core. Подробности — §9.

Базовая **форма** продукта определена: рантайм-гибрид (кит + `glue` CLI + адаптеры + визуальный клиент) с одной публичной интеграционной границей (контракт CLI) и визуализацией карты в поставке; **доставка — core-плагин `glue` (команда + карта + движок судьи + адаптеры + общие навыки) и контент-паки П1/П2/П3 с зависимостью от core; П1+П2 первыми, П3 в беклоге**; общее живёт в core в единственном экземпляре (платформенные зависимости вместо самодедупа), провенанс вето не пересекает границу контент-пака.
