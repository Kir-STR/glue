# Glue ← invoker — поглощение (design, срез 1)

**Дата:** 2026-06-25
**Статус:** рабочий design. Ведётся поверх концепции (`2026-06-23-glue-concept-design_v2.md`) и формы (`2026-06-24-glue-form-design_v6.md`); их не переписывает.
**Решение:** вариант **A** — полное слияние invoker в Glue. invoker как отдельный продукт уходит; его контент, механика и retro-цикл переезжают в Glue, который добавляет сверху то, чего у invoker нет (provenance, карта, judge).

---

## Зачем

Dogfooding MVP (П1) показал: инъекция тела правил через хук **дублирует** нативный механизм Claude Code (`CLAUDE.md` / `.claude/rules`). invoker как раз доставляет правила нативно — раскладывает файлы, которые платформа подхватывает сама. Поэтому Glue поглощает зрелую invoker-механику доставки, а свою ценность (provenance/карта/judge) наслаивает поверх.

**invoker** (исходный продукт): детерминированный скаффолдер — копирует выбранные модули-правила в проект (`.claude/rules/*.md` + `CLAUDE.md`/`AGENTS.md`/`GEMINI.md`), резолвит зависимости модулей, ведёт retro-цикл эволюции правил. Ядро — `lib/*.mjs` (`registry`/`resolve`/`blocks`/`scaffold`) с тестами, без внешних зависимостей.

## Архитектурный инвариант (несущий)

> **Файлы — источник истины.** Раскладчик материализует файлы и манифест доставки. Индексатор (срез 4) строит из них **производный** граф. Граф никогда не источник, из которого генерируются файлы.

Отсюда: раскладка и построение графа — **разные ответственности**. Появление provenance-графа в срезе 4 **не переписывает** раскладчик — индексатор лишь потребляет факты манифеста. Это снимает мнимую «двойную работу» и обосновывает перенос (A), а не переписывание (B): B неявно сделал бы граф источником, что противоречит форме («источник правды — файлы; состояние производно и пересобираемо»).

## Границы: core владеет механизмом, паки владеют контентом

**`glue-core`** (механизм раскладки):
- команда `glue init` — раскладывает выбранные модули из установленных контент-паков в проект;
- `resolve` — топосорт зависимостей модулей (**перенос из invoker + тесты**, инвариантная механика);
- `blocks` — фильтрация `<!-- module:<id> -->` в инструкц-шаблонах (**перенос + тесты**, инвариантная);
- `registry` — загрузка/валидация реестра модулей пака по **Glue-схеме** (новая, не invoker `modules.json`);
- `planner` — чистая функция: (паки, выбор, состояние диска) → **план раскладки** с разрешёнными конфликтами; никакого I/O;
- `writer` — исполняет план: пишет файлы, затем публикует **манифест доставки**.

**`glue-rules`** (контент): шаблоны модулей-правил (11 из invoker), инструкц-шаблоны (`CLAUDE.md`/`AGENTS.md`/`GEMINI.md`), реестр модулей пака.

### Декларативный контракт пака

Core **не импортирует исполняемый код** пака. Пак объявляет контракт декларативно — в своём manifest:
- `glueContract` — версия контракта пак↔core;
- относительный путь к реестру модулей (данные, не код);
- относительный путь к каталогу шаблонов правил;
- относительный путь к инструкц-шаблонам.

Core читает эти артефакты как **данные** по объявленным путям. Внутреннюю структуру пака core не знает — только контракт. (Ужесточение против invoker, где механизм и контент жили в одном пакете.)

## Поток данных

```
glue init
  → core находит установленные glue-* паки (installed_plugins.json — уже умеем из MVP)
  → читает их реестры модулей по контракту пака
  → resolve зависимостей выбранных модулей (каскад)
  → planner: план раскладки (что, откуда, куда) + разрешение конфликтов ДО записи
  → writer: запись .claude/rules/*.md + CLAUDE.md/AGENTS.md/GEMINI.md
  → writer: атомарная публикация манифеста доставки (последним)
```

### Многодвижковость

Переносится из invoker: `blocks`-фильтрация + набор движков (`claude`/`agents`/`gemini`). `glue init` принимает выбор движков; для каждого собирает инструкц-файл вырезанием невыбранных `<!-- module:<id> -->` блоков. Тексты не синтезируются — собираются из шаблонов.

## Манифест доставки

Файл `.glue/manifest.json` — **фактический результат записи**, не план.

**Заголовок:**
- `schemaVersion` — версия схемы манифеста;
- `deliveryId` — идентификатор завершённой раскладки;
- `completedAt` — время завершения (передаётся в процесс, не берётся внутри детерминируемой логики);
- `engines` — выбранные движки;
- `modules` — выбранные + разрешённые модули;
- `status: "complete"` — ставится только после успешной записи всего набора;
- `files` — записи фактически материализованных файлов.

**Запись файла:** `{ producerPack, packVersion, sourceTemplate, targetPath, writtenHash }`, где `writtenHash` — хеш содержимого **после записи**.

**Атомарность (last-known-good).** Манифест публикуется **последним**, после успешной записи всего набора. При частичном сбое старый валидный манифест **не заменяется** — система остаётся на прошлой достоверной раскладке. Это применение концептового last-known-good к доставке правил.

Назначение манифеста — **техническая трассируемость доставки** (факты для будущего индексатора, срез 4), **не** семантический provenance-граф.

## Контракт конфликтов (вычисляется planner'ом до первой записи)

Для каждого целевого файла `target`:

| Состояние | Решение |
|---|---|
| отсутствует | **записать** |
| существует, управляется прошлым манифестом, текущий хеш `==` прежнего `writtenHash` | **обновить** (файл не трогали с раскладки) |
| существует, хеш `!=` прежнего `writtenHash` | **конфликт** (правлен вручную) |
| существует, прошлому манифесту неизвестен | **конфликт** |
| любой конфликт + `--force` | **явная перезапись** |

**Planner вычисляет все решения до первого изменения.** Writer не обнаруживает ожидаемые конфликты посреди записи — он исполняет уже разрешённый план. Безопасный default — **не перезаписывать** изменённые/неизвестные файлы без `--force`. (Поправка к invoker, который перезаписывает безусловно.)

## Снятие body-injection — атомарно, через условный хук

После нативной раскладки инъекция тела правил из 0.1.1 становится дублем (две точки доставки, лишние токены, риск рассинхрона). Снимаем — но **без окна потери правил**.

SessionStart-хук прекращает инъекцию **только если выполнены ВСЕ условия**:
- `schemaVersion` манифеста поддерживается;
- `status == "complete"`;
- обязательные Claude-targets присутствуют на диске;
- их текущие хеши совпадают с `writtenHash` манифеста;
- набор относится к актуально установленным версиям паков.

**Иначе fallback остаётся** (хук инжектит тело, как в 0.1.1) **+ диагностика в stderr**. Stale / повреждённый / частичный манифест **не** отключает доставку. Это и есть атомарный переход: нативная раскладка перехватывает доставку ровно тогда, когда она доказуемо валидна — иначе работает старый путь (fail-closed + честная диагностика непокрытого).

## Перенос invoker-кода — поразрезно

| Часть | Судьба |
|---|---|
| `resolve` | перенос дословно + тесты (инвариант) |
| `blocks` | перенос дословно + тесты (инвариант) |
| `registry` | **новая Glue-схема** реестра (не invoker `modules.json`) |
| `scaffold` | **разделить** на `planner` (чистое планирование + конфликты) и `writer` (файловая запись + манифест) |
| `.invoker/modules.json`, идентичность `plugin: invoker` | **не переносим** — сразу Glue-манифест |
| templates (правила, инструкции) | в `glue-rules`, **не** в core |

### Аудит `.invoker/*` путей в переносимом контенте

Правила invoker (особенно `retro-loop`, `worktree-workflow`) ссылаются на `.invoker/` инфраструктуру. «Дословно» допустимо **только** там, где эти контракты осознанно сохраняются или мигрируют. Перед переносом каждого модуля — проверить ссылки на `.invoker/*` и решить пофайлово: сохранить, мигрировать на `.glue/`, или вынести в отложенный срез (retro-loop = срез 3). Немигрированные `.invoker/`-ссылки в Glue-контенте недопустимы.

## Порядок реализации — два PR, доставка не прерывается

1. **`glue-rules`: контракт пака + контент.** Пак объявляет `glueContract` + пути, несёт 11 модулей-шаблонов (после `.invoker/*`-аудита) и инструкц-шаблоны. Старый fallback (инъекция 0.1.1) продолжает работать — раскладчика ещё нет, манифеста нет, условие отключения не выполнено.
2. **`glue-core`: init-механизм + условное отключение fallback.** `resolve`/`blocks`/`registry`/`planner`/`writer` + команда `glue init` + условный хук. После успешного `glue init` хук перестаёт инжектить (условия выполнены); без раскладки — fallback.

Каждый промежуточный HEAD сохраняет доставку правил.

## Scope — что НЕ входит в срез 1

Осознанно отложено: карта/визуализация (срез 2), **provenance-граф (индексатор — срез 4)**, **retro-loop (срез 3)**, **judge/контроль (срез 4)**, виды артефактов как Glue-онтология. Срез 1 материализует файлы и манифест доставки; каноническая модель и граф строятся последующими слоями.

## Открытые вопросы (для следующих срезов, не блокируют срез 1)

- Точная Glue-схема реестра модулей пака (поля сверх invoker `modules.json`).
- Куда мигрирует `.invoker/` инфра retro-цикла (`.glue/`?) — решается в срезе 3.
- Формат выбора модулей в `glue init` (интерактив / флаги / профиль) — детализируется в плане.
