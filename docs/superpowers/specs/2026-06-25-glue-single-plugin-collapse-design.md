# Glue — схлопывание в один плагин (design, greenfield)

**Дата:** 2026-06-25
**Статус:** рабочий design. Замещает решения о доставке из формы и спеки поглощения (см. «Что замещает»). Концепцию (`2026-06-23-glue-concept-design_v2.md`) не переписывает: пять видов, провенанс, карта, judge — остаются как онтология; меняется только **упаковка доставки**.
**Решение:** один плагин `glue` со встроенным контентом; **greenfield-переписывание**, не миграционный рефактор. Маркетплейс-распространение курируемого контента третьими лицами **снято сознательно** при низком adoption.

---

## Что замещает (supersedes)

Существующие документы — read-only исторические артефакты, **не правятся**. Этот design замещает их в части доставки:

- `2026-06-24-glue-form-design_v6.md` § «Декомпозиция доставки — core-плагин + контент-паки», § «Совместимость версий — нативная», § «Провенанс не пересекает границу контент-пака» — в части механизма доставки (трёхпаковая модель → один плагин);
- `2026-06-25-glue-invoker-absorption-design.md` § «Границы: core владеет механизмом, паки владеют контентом», § «Декларативный контракт пака» (как **меж-плагинный** контракт), § «Идентичность модулей между паками», § «Порядок реализации — два PR».

Сохраняются без изменений: концептовая онтология видов, инвариант «файлы — источник правды», roadmap срезов 2-4 (карта, retro-loop, provenance-граф+judge) — они лишь теряют меж-паковые допущения.

## Зачем greenfield, а не рефактор

Отгруженный код (`glue-core` 0.2.2 + `glue-rules` 0.2.1) несёт dogfooding-наследие: инлайн-дубль fallback-ридера в `bin/glue.mjs`, MVP-комментарии («Task 2.8 may modify»), body-injection 0.1.1 как основной путь, меж-плагинный скан `installed_plugins.json` в двух местах. При низком adoption чистый плагин дешевле и честнее серии миграционных рефакторов.

**Критерий переписывания:** переносим **наблюдаемое поведение и инварианты**, а не старую внутреннюю декомпозицию. Старые `glue-core`/`glue-rules` — **референс поведения**, не база нового кода.

## Снятые инварианты (greenfield от них свободен)

- скан `installed_plugins.json` и кросс-папочное чтение контента чужого плагина (несущий риск №1 формы — **исчезает**, контент в своей папке);
- `dependencies` между плагинами, version-constraints, git-теги `{plugin}--v{version}`, auto-delivery — вся платформенная машинерия формы v6;
- квалифицированные `pack:module` ID, межпаковые зависимости, fail-fast на коллизию ID между паками (`mergePackRegistries`);
- совместимость со старыми манифестами и именами плагинов `glue-core`/`glue-rules`.

## Сохранённые инварианты (доказали ценность)

- **конфликт-детекция** — не перезаписывать правленные вручную файлы без `--force`;
- **TOCTOU-защита** — перепроверка хеша непосредственно перед записью;
- **атомарность манифеста** — публикуется последним; при частичном сбое не заменяется; recovery смешанного состояния через хук-валидацию + повторный `init`;
- **resolve зависимостей** — топосорт `dependsOn` (реально используется: `pr-policy`→`worktree-workflow`);
- **path-safety target-валидации** — `target` остаётся внутри проекта и разрешённой зоны (`.claude/`, `CLAUDE.md`/`AGENTS.md`/`GEMINI.md`, `.glue/`); source-safety растворяется (свой bundle), target-safety остаётся;
- **мультидвижок** — `blocks`-фильтрация `<!-- module:id -->` для `claude`/`codex`(→`AGENTS.md`)/`gemini`;
- **честная диагностика непокрытого + fail-closed** — stale/частичный манифест не отключает доставку.

## Acceptance-сценарии (поведенческий контракт переписывания)

Эти сценарии — то, что новый плагин обязан воспроизвести. Зелёные acceptance-тесты = условие cutover'а legacy.

1. **`/glue:list`** — показать доступные модули (id, title, group, default, note).
2. **`/glue:init`** — выбрать движки + модули, материализовать `.claude/rules/*.md` + инструкц-файлы, опубликовать манифест.
3. **`/glue:status`** — доказать состояние доставки по **всем заявленным движкам** (что записано, совпадает ли с диском, есть ли более новая версия контента).
4. **SessionStart** — проверить native delivery; **fallback с явной семантикой** (см. ниже).
5. **Повторный `init`** — идемпотентен: уже записанные файлы (`текущий == plannedHash`) не объявляются конфликтом.
6. **Правленный пользователем файл** — конфликт, без молчаливой перезаписи (только `--force`).
7. **Снятый модуль** — безопасное удаление только **собственного неизменённого** файла (правленный → конфликт).
8. **`codex` в выборе движков** — создаётся `AGENTS.md` (через `blocks`-фильтрацию).
9. **Неизвестный движок** — отклоняется **до любой записи** (fail-fast).
10. **`manifest.engines`** — содержит только **реально доставленные** движки (честный манифест, не запрошенный набор).

### Fallback-семантика (решено: R1)

**Gate валидности (узкий, для решения inject↔молчать)** — только **обязательные Claude-targets** (`CLAUDE.md` + `.claude/rules/*`), потому что fallback инжектит в Claude-контекст:
- **native валиден** = `status: complete` + обязательные Claude-targets на диске + их хеши совпадают с манифестом. **Версия в gate не участвует** (см. «Манифест»). → хук **молчит**: правила лежат нативно, Claude читает их сам. В стационаре инъекции нет, токен-дубля нет.
- **native невалиден** → fallback инжектит тела правил из своего `content/` + диагностика в stderr («native delivery не активна — запусти `/glue:init`»).

> Полнота по **всем** движкам (есть ли `AGENTS.md`/`GEMINI.md`) — забота `/glue:status`, **не** Claude-gate'а. Иначе отсутствие `AGENTS.md` либо ложно включит Claude-fallback (хотя Claude в порядке), либо останется невидимым.

**Какие модули инжектировать в fallback** (явно, чтобы не активировать лишнее):
- манифест **читаем** (даже hash-stale/version-stale) → инжектить его `modules`, разрешённые через `resolve`;
- манифест **отсутствует/нечитаем/corrupt/неподдерживаемый `schemaVersion`** → resolved **defaults** (`default: true` + их `dependsOn`);
- **никогда не инжектить все модули неявно** — это активировало бы невыбранные правила.

Обоснование R1 (а не «только диагностировать»): модель продукта — защита-по-умолчанию + fail-closed. Нулевая видимость правил до первого `init` — реальная дыра для guardrail. Стоимость инъекции ограничена деградированным состоянием (до `init` / recovery смешанного состояния); в стационаре её нет. Это условная развилка спеки поглощения, пере-направленная с чужого пака на свой bundle.

## Структура плагина

```
plugins/glue/
  .claude-plugin/plugin.json        # name: "glue"
  glue.contract_v1.json             # внутренний bundle-контракт (версия в имени, versioning.md)
  hooks/hooks.json                  # SessionStart → bin/glue.mjs session-start
  bin/glue.mjs                      # тонкий диспетчер подкоманд над src/
  src/
    bundle.mjs                      # загрузка своего bundle (вместо discovery; без скана реестра)
    resolve.mjs                     # топосорт dependsOn (перенос поведения)
    blocks.mjs                      # вырезание <!-- module:id --> для движков (нужен сценарию 8)
    plan.mjs                        # чистое планирование + конфликт-контракт + path-safety target
    apply.mjs                       # запись файлов + TOCTOU + публикация манифеста
    status.mjs                      # отчёт доставки + staleness
    manifest.mjs                    # формат/чтение/запись манифеста
  content/
    bundle.json                     # реестр модулей (вместо registry.json)
    modules/                        # шаблоны правил (вместо templates/)
    instructions/                   # CLAUDE.md.tmpl / AGENTS.md.tmpl / GEMINI.md.tmpl
  test/
```

**Дисциплина «контент как данные»** сохраняется и внутри одного плагина: `src/` не импортирует контент как код — читает `content/` по путям из bundle-контракта.

### Внутренний bundle-контракт

`glue.contract_v1.json` — **внутренний** контракт раскладки (не provider API для внешних паков). Версия — **в имени файла** (`versioning.md`: «версию в содержимом не дублируем»), поэтому поля `contractVersion` в теле нет. Без поля `kind` (нет внешних паков):

```json
{
  "registry": "content/bundle.json",
  "modulesDir": "content/modules",
  "instructionsDir": "content/instructions"
}
```

Польза шва: механизм не знает расположение контента; контент — тестируемые данные; будущие виды (Решение/Ограничение) добавляются без переделки loader'а.

### Формат реестра (`content/bundle.json`)

Переносится поведение текущего реестра: `module-id → { title, group, default, note, templates[], instructionBlock, dependsOn[] }` (имена в `templates[]` резолвятся относительно `modulesDir`). Решения/Ограничения как наполненные виды — **не в этом срезе** (онтология слоёв 4+); сейчас реестр несёт виды Правило/Знание (10 модулей invoker без `retro-loop` — он завязан на retro-инфру среза 3).

### Манифест

Формат свежий, без back-compat с манифестами эпохи `glue-core`. Поле `producerPack` **сохраняется** со значением `"glue"`, `packVersion` = версия плагина из `plugin.json`.

**Валидность развязана от версии (решено).** Gate валидности (native↔fallback) считает только: `status: complete` + хеши обязательных Claude-targets совпадают с манифестом. **Bump версии плагина сам по себе layout не инвалидирует** — иначе каждый релиз кода форсил бы ре-`init` без изменения контента, что противоречит «файлы — источник правды, состояние производно». `packVersion` остаётся **чистым провенансом** (кто записал); в бинарный gate он не входит. (Уточнение прежней позиции: поле живёт, но выходит из гейта валидности.)

**Сигнал «доступна новая версия контента» — по хешам, не по `packVersion`.** `/glue:status` вычисляет это сравнением `plannedHash` модулей текущего bundle с `writtenHash` манифеста (и диском), а не сравнением версий. Так сигнал отражает фактическое расхождение контента, а не релиз кода.

## Порядок поставки — вертикальные зелёные срезы

Greenfield живёт в `plugins/glue/` **параллельно** нетронутым `glue-core`/`glue-rules`. Каждый срез — зелёный HEAD. **Срез ≠ один PR:** срез может занимать несколько PR, если иначе пробивает бюджет.

0. **импорт контента** — отдельный механический PR: `content/bundle.json` + `modules/` (10 модулей invoker без `retro-loop`) + `instructions/` (3 шаблона). ~14 файлов, поэтому **сам по себе** PR — иначе с кодом среза 1 пробьёт 15-файловый cap;
1. **каркас + `bundle`/`list`** — плагин-скелет, `bundle`-loader (своя папка), `/glue:list`;
2. **`plan`/`apply`/`manifest`** — материализация + конфликт-контракт + TOCTOU + атомарный манифест;
3. **`status` + хук** — отчёт доставки (все движки) + SessionStart с решённой fallback-семантикой (узкий Claude-gate);
4. **skills + интеграционный сценарий** — три SKILL.md + сквозной прогон **10** acceptance-сценариев;
5. **marketplace cutover** — удалить `glue-core`/`glue-rules`, переключить маркетплейс на `glue`.

**Бюджет.** Каждый PR отдельно укладывается в `pr-policy` (target 400, hard-cap 800 строк / 15 файлов reviewable). Импорт контента (~14 файлов, byte-identical перенос поведения) вынесен в PR 0 именно из-за файло-cap'а; код-срезы остаются узкими.

### Cutover и миграция существующего проекта

**Cutover (срез 5) — деструктивный класс:** отдельный operator-gate (буквальное «да»), только после зелёных acceptance (срез 4). До cutover'а legacy продолжает доставлять — окна потери нет.

**Два хука одновременно.** Снятие записи из marketplace **не** удаляет уже установленные legacy-плагины → их SessionStart-хук продолжит работать рядом с новым. Поэтому cutover **деинсталлирует** `glue-core`/`glue-rules` (не только de-list), а acceptance проверяет: активен **ровно один** glue SessionStart-хук.

**Legacy-манифест в управляемом проекте — безопасная деградация, без bespoke-команды.** Bespoke migration-команда при текущем adoption — YAGNI; безопасный путь даёт уже описанная машинерия:
- legacy `.glue/manifest.json` (чужой `schemaVersion`/`producerPack`) → новый `nativeDeliveryValid` обязан трактовать неподдерживаемый `schemaVersion` как **невалидный** (не падать) → fallback R1;
- новый `glue init`: byte-identical legacy-файлы → строка конфликт-контракта «`текущий == plannedHash` → уже записан → пропустить» (idempotency, без миграции); разошедшиеся → **конфликт** (не молчаливая перезапись);
- **этот dev-репо** не glue-managed (нет `.glue/`, правила рукописные) → мигрировать нечего; throwaway тест-проект — пересоздать с нуля.

Это и есть осознанно выбранный migration flow: deinstall legacy + один хук + безопасная деградация неподдерживаемого манифеста + конфликт-контракт. Отдельная команда не вводится.

> **Имплементеру явно:** legacy-выбор модулей **не сохраняется** автоматически. Новый runtime **не читает** legacy-манифест ради «умной» миграции (никакого чтения `producerPack: "glue-rules"`-манифеста). При неподдерживаемом манифесте пользователь заново выбирает модули в новом `/glue:init`; до этого fallback берёт **defaults**. Это сознательно — не повод писать legacy-ридер.

## Scope — что НЕ входит

Осознанно отложено (roadmap концепта/поглощения, не отменяется): карта/визуализация (срез 2 поглощения), retro-loop (срез 3), provenance-граф + judge/контроль (срез 4), Решения/Ограничения как наполненные виды. Этот design — **переписывание доставки П1** (Правила/Знания) в один плагин; каноническая модель и граф строятся последующими слоями поверх.
